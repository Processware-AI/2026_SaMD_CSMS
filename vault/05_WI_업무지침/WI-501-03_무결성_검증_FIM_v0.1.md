# 무결성 검증(FIM) 업무지침 (WI-501-03)

> 상위 절차: [[PRO-MDCS-501_보안_모니터링_및_로그_관리_절차_v1.0]] · §5 단계 3 (무결성 검증)

## 1. 업무 목적

FIM(File Integrity Monitoring) 도구를 통해 운영 환경의 핵심 파일·설정·바이너리의 무결성을 상시 검증하고, 인가되지 않은 변경을 즉각 탐지·알림·대응함으로써 악성코드 삽입·설정 조작·백도어 설치로 인한 보안 침해를 사전에 차단한다.

## 2. 수행 주체

- **주 수행자**: SecOps 팀 (FIM 도구 운영·알림 처리·대응)
- **검토자**: 인프라팀 (변경 인가 여부 확인), 개발팀 (의료기기 바이너리 변경 인가 확인)
- **승인자**: 오동석 (FIM 대상 파일 목록 최초 승인, 주요 예외 승인)

## 3. 범위

- 운영 서버 OS 시스템 바이너리 및 설정 파일 (`/etc`, `/bin`, `/sbin`, `/usr/bin` 등)
- 웹 서버·WAS·데이터베이스 서버 설정 파일 및 배포 파일
- 의료기기 펌웨어 이미지 및 부트로더 (적용 가능한 범위)
- 보안 도구·감사 로그 설정 파일 (SIEM 에이전트, AV 설정 등)
- 인증서 및 암호화 키 저장 경로

## 4. 입력 자료 / 산출물

- **Input**
  - 자산 목록 및 하드닝 완료 시스템 목록
  - FIM 대상 파일·디렉터리 목록 (초기 기준선)
  - 인가된 변경 일정 (패치·배포 계획, 변경 관리 시스템 연동)
  - [[TMP-OPS-020_FIM_모니터링_대장_v0.1]] (초기 해시·서명 등록)
- **Output**
  - FIM 알림 이벤트 로그
  - [[TMP-OPS-021_FIM_알림_처리_기록_v0.1]] (탐지→분석→조치 이력)
  - 월간 FIM 운영 현황 보고서 (오탐율·미처리 건수·대상 커버리지)
  - 비인가 변경 확인 시 PRO-MDCS-502 연계 IR 티켓

## 5. 수행 절차

### 5.1 사전 준비

1. SecOps 팀은 FIM 도구(Tripwire, OSSEC, AIDE, Wazuh FIM 모듈 등)가 대상 시스템에 설치·활성화되어 있는지 확인하고, 에이전트 버전을 최신으로 유지한다.
2. [[TMP-OPS-020_FIM_모니터링_대장_v0.1]]에 모니터링 대상 파일·디렉터리 목록, 최초 등록 날짜, 등록자를 기재하고 오동석 승인을 받는다.
3. 변경 관리 시스템(ITSM)과 FIM 도구 간 연동 설정을 확인하여, 승인된 변경 시간대에는 알림 억제(Maintenance Window) 또는 예외 처리가 자동 적용되도록 구성한다.
4. FIM 알림 수신자 목록(SecOps 온콜·SIEM 연동)을 최신 상태로 유지하고, 알림 채널(이메일·SIEM 대시보드·모바일 푸시)을 사전 테스트한다.

### 5.2 수행 단계

1. **기준선(Baseline) 초기화 및 갱신**
   - 하드닝 완료 직후 또는 인가된 대규모 변경(OS 업그레이드, 주요 패치) 후 FIM 기준선을 재초기화한다.
   - 기준선 초기화는 반드시 오동석 또는 위임받은 SecOps 팀장의 승인 하에 수행하고, 초기화 사유와 수행자를 [[TMP-OPS-020_FIM_모니터링_대장_v0.1]]에 기록한다.

2. **실시간·정기 무결성 검증**
   - FIM 도구를 실시간 감시 모드로 운영하여 파일 생성·수정·삭제·권한 변경 이벤트를 즉시 탐지한다.
   - 실시간 감시가 기술적으로 불가한 시스템(임베디드 장치 등)은 최소 1일 1회 정기 스캔을 수행한다.
   - 의료기기 펌웨어는 배포 후 72시간 이내 해시 검증 스캔을 수행한다.

3. **알림 수신 및 1차 분류**
   - FIM 알림 수신 시 SecOps 담당자는 30분 이내 [[TMP-OPS-021_FIM_알림_처리_기록_v0.1]]을 생성한다.
   - 변경 관리 시스템(ITSM)에서 동 시간대 승인된 변경 티켓이 존재하는지 확인한다.
     - 티켓 존재 → "인가 변경"으로 분류하고 기준선 갱신 계획을 수립한다.
     - 티켓 없음 → "비인가 변경 의심"으로 분류하고 다음 단계(4)로 진행한다.

4. **비인가 변경 의심 시 심층 분석**
   - 변경된 파일의 내용·타임스탬프·실행 권한·디지털 서명을 확인한다.
   - 프로세스 실행 로그와 교차 분석하여 변경 행위자(사용자 계정·프로세스)를 추적한다.
   - 악성 코드 삽입 또는 백도어 설치로 판단되는 경우, 즉시 PRO-MDCS-502(침해사고 대응)를 트리거한다.

5. **인가 변경 후 기준선 업데이트**
   - 인가된 변경으로 확인된 파일에 대해 변경 완료 후 FIM 기준선에 반영(재서명·재해시)한다.
   - 기준선 업데이트 내역을 [[TMP-OPS-020_FIM_모니터링_대장_v0.1]]에 기록한다.

6. **월간 운영 현황 보고**
   - 매월 말 SecOps 팀장은 FIM 알림 건수·오탐율·미처리 건수·대상 커버리지를 집계하여 오동석에 보고한다.
   - 오탐율이 10% 초과 시 FIM 규칙을 재검토하고 튜닝 계획을 수립한다.

### 5.3 완료 조건

- [ ] 모든 대상 시스템이 [[TMP-OPS-020_FIM_모니터링_대장_v0.1]]에 등록되어 있고 오동석 승인이 확인된다
- [ ] FIM 알림 발생 건 전수에 대해 [[TMP-OPS-021_FIM_알림_처리_기록_v0.1]]이 작성되었다
- [ ] 비인가 변경 의심 건이 4시간 이내 분류·처리되었거나 PRO-MDCS-502로 에스컬레이션되었다
- [ ] 인가된 변경 후 FIM 기준선 업데이트가 완료되었다
- [ ] 월간 운영 현황 보고서가 오동석에 제출되었다
- [ ] 실시간 감시 불가 시스템에 대해 1일 1회 이상 정기 스캔이 실행되고 있음이 확인된다

## 6. 인터페이스 부서

- **인프라팀**: 변경 인가 여부 확인, FIM 에이전트 설치·네트워크 환경 지원
- **개발팀**: 의료기기 바이너리·펌웨어 인가 변경 통보, 해시 값 제공
- **오동석**: FIM 대상 목록 승인, 기준선 재초기화 승인, 주요 비인가 변경 보고 수신
- **CSIRT**: 비인가 변경 확인 시 침해사고 대응 연계
- **변경관리위원회(CAB)**: 승인된 변경 일정 제공 (Maintenance Window 정보)

## 7. 주의사항 / 예외 처리

### 7.1 고빈도 변경 경로의 오탐 관리

로그 파일 경로(`/var/log/`), 임시 파일 경로(`/tmp/`), 데이터베이스 데이터 파일 등 업무상 빈번하게 변경되는 경로를 FIM 감시 대상에 포함할 경우 오탐이 폭증한다. 이러한 경로는 FIM 감시 대상에서 제외하되, 접근 로그 기반 모니터링(SIEM 연계)으로 대체 통제를 적용한다. 제외 경로는 [[TMP-OPS-020_FIM_모니터링_대장_v0.1]]에 명시하고 오동석 승인을 받는다.

### 7.2 패치 배포 시 일괄 기준선 갱신

OS 패치·소프트웨어 업데이트로 다수의 시스템 파일이 동시에 변경되는 경우, 패치 창(Maintenance Window) 시간 동안 FIM 알림을 억제(Suppress)하고 패치 완료 후 일괄 기준선 갱신을 수행한다. 억제 기간은 ITSM 변경 티켓 기재 시간을 기준으로 자동 설정하며, 최대 4시간을 초과하지 않는다. 억제 기간 중 발생한 변경 이벤트는 패치 완료 후 전수 검토한다.

### 7.3 의료기기 펌웨어 FIM 제약

의료기기 특성상 실시간 에이전트 설치가 불가한 경우, 네트워크 분리된 격리 환경에서 업무 전·후 이미지 해시 비교 방식으로 무결성을 검증한다. 해시 알고리즘은 SHA-256 이상을 사용하고, 비교 결과는 [[TMP-OPS-021_FIM_알림_처리_기록_v0.1]]에 준하는 형식으로 문서화한다.

### 7.4 FIM 도구 자체 보안

FIM 도구 설정 파일 및 기준선 데이터베이스 자체가 변조될 위험이 있다. FIM 데이터베이스는 읽기 전용 마운트 또는 분리된 보안 스토리지에 보관하고, FIM 에이전트 설정 변경 권한은 SecOps 팀장으로 제한한다.
