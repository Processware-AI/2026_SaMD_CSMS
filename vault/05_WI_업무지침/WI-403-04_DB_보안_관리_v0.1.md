# DB 보안 관리 업무지침 (WI-403-04)

> 상위 절차: [[PRO-MDCS-403_데이터_및_파일_보안_관리_절차_v1.0]] · §5 단계 5

## 1. 업무 목적

데이터베이스에 대한 접근을 최소권한·업무분리 원칙에 따라 통제하고, 모든 쿼리 실행 이력을 로깅하며, 민감 정보(개인정보·의료데이터·인증정보)가 저장된 컬럼에 암호화를 적용함으로써 내부자 오남용·외부 침해·SQL Injection 등으로 인한 데이터 유출을 예방한다. SaMD-CSMS 제08조 제1호의 DB 보안 요건(MDCS-R-081) 및 제08조 제2호의 비식별화 요건(MDCS-R-082)을 충족한다.

## 2. 수행 주체

- **주 수행자**: DBA (계정 관리, 권한 설정, 암호화 설정, 쿼리 로그 관리)
- **검토자**: SecOps (이상 쿼리 탐지, 권한 검토 연계)
- **승인자**: 오동석 (관리자 계정·예외 승인), 라인매니저 (일반 계정 권한 승인)
- **협력자**: DPO (민감 컬럼 분류, 비식별화 요건 정의)

## 3. 범위

- 운영 환경의 모든 관계형 DB(MySQL, PostgreSQL, Oracle, MSSQL 등) 및 NoSQL DB(MongoDB, Elasticsearch 등)
- 개발·스테이징 환경 DB (운영 데이터 복사본 포함 시 동일 통제 적용)
- DB 계정 유형: 애플리케이션 서비스 계정, DBA 관리 계정, 배치 처리 계정, BI/리포팅 읽기 전용 계정
- 민감 데이터 컬럼: 개인식별정보(성명·주민번호·연락처), 인증정보(비밀번호·토큰), 의료정보(진단코드·처방정보), 결제정보

## 4. 입력 자료 / 산출물

- **Input**
  - [[TMP-DBA-001_DB_계정_신청서_v0.1]] (신청자·소속·업무 목적·필요 권한·기간·대상 DB)
  - 데이터 민감도 분류 결과 (DPO 산출, PRO-MDCS-403 §5 단계 1)
  - DB 스키마 및 데이터 사전 (민감 컬럼 식별용)
- **Output**
  - DB 계정 생성·변경·삭제 기록
  - [[TMP-DBA-002_DB_권한_검토_대장_v0.1]] (분기 권한 검토 결과)
  - [[TMP-DBA-003_민감컬럼_암호화_현황표_v0.1]] (암호화 적용 컬럼·알고리즘·키 참조)
  - 쿼리 감사 로그 (DB Native Audit 또는 외부 DAP 산출물)
  - 미사용 계정 비활성화 기록

## 5. 수행 절차

### 5.1 사전 준비

1. DBA는 DPO와 협력하여 데이터 사전을 기준으로 민감 컬럼을 식별하고 [[TMP-DBA-003_민감컬럼_암호화_현황표_v0.1]] 를 작성한다. 민감 컬럼 목록은 스키마 변경 시 즉시 업데이트한다.
2. DB 인스턴스별 기본 관리자 계정(root, sa, sys, admin 등)의 비밀번호를 최소 16자 이상 복잡도 비밀번호로 변경하고, 기본 계정 비활성화 여부를 점검한다.
3. 애플리케이션 서비스 계정과 DBA 관리 계정의 권한을 분리하며, 서비스 계정이 DDL(CREATE, DROP, ALTER) 권한을 갖지 않도록 초기 설정한다.
4. DB Audit 기능(MySQL Audit Plugin, Oracle Unified Auditing, PostgreSQL pgaudit 등)이 활성화되고 로그가 외부 SIEM으로 전송되는지 확인한다.

### 5.2 수행 단계

1. **DB 계정 생성 및 권한 부여**
   - 신청자가 [[TMP-DBA-001_DB_계정_신청서_v0.1]] 를 작성한다. 필수 항목: 신청자·소속·업무 목적·대상 DB 인스턴스·스키마·필요 권한(SELECT/INSERT/UPDATE/DELETE/EXECUTE)·사용 기간.
   - 라인매니저가 업무 필요성을 승인하며, DBA 관리 계정 또는 개인정보 DB 접근 계정은 오동석 추가 승인을 받는다.
   - DBA가 최소권한 원칙에 따라 필요한 스키마·테이블·작업에 한정하여 권한을 부여하고, 권한 부여 쿼리(GRANT 문)를 계정 신청서에 기재한다.
   - 계정 생성 직후 초기 비밀번호 강제 변경 플래그를 설정하며, 비밀번호 정책(최소 12자, 복잡도, 90일 만료)을 적용한다.
2. **민감 컬럼 암호화 적용**
   - [[TMP-DBA-003_민감컬럼_암호화_현황표_v0.1]] 에 따라 개인식별정보·인증정보 컬럼에 투명 데이터 암호화(TDE) 또는 컬럼 레벨 암호화(AES-256-GCM)를 적용한다.
   - 인증정보(비밀번호) 컬럼은 단방향 해시(bcrypt cost≥12 또는 Argon2id)를 적용하고, 평문 저장을 엄격히 금지한다.
   - 암호화 키는 [[WI-402-05_HSM_KMS_운영_v0.1]] 에 따라 KMS에서 관리하며, DB 서버에 키를 직접 저장하지 않는다.
   - 암호화 적용 후 애플리케이션 암복호화 동작을 QA 테스트 환경에서 검증한다.
3. **쿼리 로깅 및 이상 탐지**
   - 모든 관리자 계정 세션의 DDL·DCL 쿼리를 전량 감사한다.
   - 개인정보 테이블에 대한 SELECT 쿼리(대량 조회 포함)는 반환 건수 임계값(예: 단일 쿼리 500건 초과) 설정을 통해 SecOps에 자동 알림을 발송한다.
   - 업무 시간 외 쿼리, 비정상 IP 접근, 실패 로그인 반복은 DAP(Database Activity Proxy) 또는 SIEM 룰로 탐지한다.
   - 감사 로그는 변조 방지를 위해 DB 로컬 저장소가 아닌 중앙 SIEM에 실시간 전송하며, 최소 1년간 보존한다.
4. **서비스 계정 관리**
   - 서비스 계정 비밀번호는 Secret Manager(Vault, AWS Secrets Manager 등)에서 관리하며, 애플리케이션 설정 파일에 평문 기재를 금지한다.
   - 서비스 계정 비밀번호는 분기 1회 또는 계정 침해 의심 시 즉시 교체하고 로테이션 기록을 남긴다.
5. **분기 권한 검토**
   - DBA가 분기 1회 전 계정 목록과 권한을 [[TMP-DBA-002_DB_권한_검토_대장_v0.1]] 에 추출한다.
   - 라인매니저 또는 시스템 소유자가 계정별 유지·회수·축소 여부를 기재하고 서명한다.
   - 회수·축소 결정은 2주 이내 적용하며, 완료 일시를 대장에 기재한다.
6. **미사용 계정 및 기본 계정 관리**
   - 90일 이상 로그인 기록이 없는 계정을 매월 탐지하여 DBA가 라인매니저에게 확인을 요청한다. 사유 없이 미사용 확인 시 비활성화한다.
   - DB 인스턴스의 기본 내장 계정(test, anonymous 등) 은 사용 여부와 관계없이 비활성화 또는 삭제한다.

### 5.3 완료 조건

- [ ] 모든 DB 계정이 [[TMP-DBA-001_DB_계정_신청서_v0.1]] 에 근거하여 생성되었다
- [ ] 민감 컬럼 암호화가 [[TMP-DBA-003_민감컬럼_암호화_현황표_v0.1]] 와 일치하여 적용되었다
- [ ] 인증정보 컬럼이 단방향 해시로 저장되어 있음이 확인되었다
- [ ] DB Audit 로그가 SIEM에 전송되고 이상 탐지 룰이 활성화되었다
- [ ] 분기 권한 검토가 완료되고 회수 결과가 2주 이내 적용되었다
- [ ] 서비스 계정 비밀번호가 Secret Manager에서 관리되고 설정 파일 평문 기재가 없다
- [ ] 미사용 계정·기본 계정이 비활성화되었다

## 6. 인터페이스 부서

- **라인매니저**: DB 계정 권한 승인·회수 결정
- **오동석**: DBA 관리 계정 및 개인정보 DB 접근 계정 추가 승인, 예외 승인
- **DPO**: 민감 컬럼 분류 정의, 개인정보 접근 감사 연계
- **SecOps**: 쿼리 감사 로그 이상 탐지, DB 침해 사고 대응
- **개발팀**: 서비스 계정 사용 요청, 암호화 API 통합 구현
- **인프라팀**: DB 서버 OS 보안 설정, 네트워크 접근통제(방화벽 룰)

## 7. 주의사항 / 예외 처리

### 7.1 긴급 장애 대응 시 관리자 계정 임시 사용

DB 장애 복구 등 긴급 상황에서 관리자 계정 직접 접근이 필요한 경우, 오동석 구두 승인 후 DBA가 Break Glass 관리자 계정을 사용한다. 사용 후 24시간 이내 해당 세션의 쿼리 로그를 오동석에게 제출하고, 사후 계정 비밀번호를 즉시 변경한다.

### 7.2 개발·스테이징 환경에서 운영 데이터 복사 시

테스트 목적으로 운영 DB 데이터를 개발·스테이징 환경에 복사하는 경우, DPO 승인을 받은 후 반드시 개인정보 마스킹 또는 가명처리([[WI-403-05_개인정보_비식별화_v0.1]] 연계)를 수행하고 복사한다. 마스킹되지 않은 운영 데이터를 개발 환경에 복사하는 것은 금지한다.

### 7.3 대량 데이터 추출 요청 (BI·분석 목적)

BI·데이터 분석 목적의 대량 데이터 추출(100만 건 초과)은 별도 데이터 제공 신청서를 제출하고 DPO·오동석 공동 승인을 받는다. 추출 파일은 암호화 전송하고, 분석 완료 후 30일 이내 안전 삭제 확인서를 제출한다.

### 7.4 암호화 키 교체 시 기존 데이터 재암호화

컬럼 레벨 암호화 키를 교체하는 경우, 기존 키로 암호화된 데이터를 신규 키로 일괄 재암호화해야 한다. 재암호화는 서비스 영향을 최소화하기 위해 배치 처리로 수행하며, 완료 전까지는 구 키와 신 키를 동시에 운용하는 키 롤오버(key rollover) 기간을 최대 30일로 제한한다.
