# 모의 침투 테스트 업무지침 (WI-301-05)

> 상위 절차: [[PRO-MDCS-301_보안_SDLC_절차_v1.0]] · §5 단계 9

## 1. 업무 목적

SaMD-CSMS 제14조 제2호에 따라 디지털의료기기 소프트웨어 및 인프라에 대해 **실제 공격자의 관점**에서 OWASP 기반 시나리오를 활용하여 모의 침투 테스트(Penetration Testing)를 수행하고, SAST·DAST·퍼징으로 탐지되지 않은 복합적·맥락적 취약점을 식별하며, 발견된 취약점의 추적·조치·검증을 체계적으로 관리한다.

## 2. 수행 주체

- **주 수행자**: QA (내부 펜테스트 실행·조율), 외부 전문 보안 업체 (연 1회 이상 외부 펜테스트)
- **지원**: R&D (환경 구성·코드 접근 지원), PSO (보안 영향 판단·범위 승인)
- **테스트 범위 승인**: PSO, VP of R&D
- **결과 검토자**: PSO, PSC
- **승인자**: VP of R&D

## 3. 범위

- **필수 수행 (릴리스 전)**:
  - 웹 애플리케이션·API 엔드포인트 (인증·권한 부여·입력 검증·세션 관리)
  - 네트워크 통신 계층 (TLS 설정·포트·프로토콜)
  - 관리자 인터페이스 및 원격 접근 경로
- **연 1회 이상 (외부 전문기관)**:
  - 전체 시스템 수준 블랙박스·그레이박스 펜테스트
  - 물리·네트워크·애플리케이션 계층 통합 평가
- **OWASP 기준 적용**:
  - OWASP Web Application Security Testing Guide (WSTG)
  - OWASP API Security Top 10
  - OWASP Medical Device Security Testing (해당 시)
- 시판 후 중대 아키텍처 변경 또는 WI-201-05 중대 트리거 발생 시 임시 펜테스트

## 4. 입력 자료 / 산출물

- **Input**
  - 펜테스트 범위 승인서 (TMP-TEST-005A 계획서)
  - 최신 아키텍처 도면·네트워크 다이어그램 (범위 정의용)
  - SAST·DAST·퍼징 리포트 (기존 취약점 현황)
  - STRIDE 위협 모델링 결과 (공격 시나리오 참고)
  - 이전 펜테스트 결과 및 미조치 항목 목록 (재검증용)
- **Output**
  - 펜테스트 계획서 (TMP-TEST-005A): 범위·방법론·일정·역할·비상 연락처
  - 펜테스트 리포트 (TMP-TEST-005): 발견 취약점·재현 방법·위험 등급·권고 조치
  - 취약점 추적 대장 업데이트 (TMP-RISK-006 통합)
  - 재검증(Retest) 결과 보고서

## 5. 수행 절차

### 5.1 사전 준비

1. **테스트 계획 수립**: PSO 와 QA 가 협력하여 TMP-TEST-005A(펜테스트 계획서)를 작성한다. 계획서에는 다음 내용을 포함한다.
   - 테스트 범위(In-scope 시스템·IP·URL·API 목록), Out-of-scope 명확화
   - 테스트 유형(블랙박스/그레이박스/화이트박스) 및 방법론
   - 일정 및 단계별 마일스톤
   - 비상 연락처 (테스트 중 서비스 장애 발생 시 즉시 연락할 담당자)
   - 데이터 처리 지침 (테스트 중 취득한 환자 데이터·자격증명 처리 방법)
2. **범위 승인**: 계획서를 PSO·VP of R&D 가 서명 승인한다. 외부 업체 수행 시 NDA(비밀유지계약)와 보안 업체 자격 확인을 선행한다.
3. **환경 준비**: 프로덕션과 동일 구성의 격리된 테스트 환경을 구성한다. 실제 환자 데이터가 포함되지 않도록 익명화 또는 합성 데이터를 사용한다.
4. **기준선 스냅샷**: 테스트 전 시스템 상태(로그·설정·DB 스냅샷)를 기록하여 테스트 후 복구 기준점으로 활용한다.

### 5.2 수행 단계

#### A. 정보 수집 단계 (Reconnaissance)

1. **수동 정보 수집**: 공개 정보(DNS·WHOIS·공개 API 문서·GitHub 코드·Job 포스팅)를 통한 공격 표면 파악.
2. **포트·서비스 스캐닝**: 네트워크 수준 서비스 노출 확인 (nmap 등). 계획서에 정의된 범위 내에서만 수행.
3. **인증·세션 분석**: 로그인 메커니즘·세션 토큰·쿠키 설정·JWT 클레임을 검토.

#### B. 취약점 탐색 단계 (Vulnerability Identification)

1. **OWASP WSTG 체크리스트 적용**: 다음 10개 범주를 체계적으로 검증한다.
   - OTG-INFO: 정보 수집
   - OTG-CONFIG: 설정 관리 (TLS 버전·헤더·기본 자격증명)
   - OTG-AUTHN: 인증 (무차별 대입·자격증명 재사용·MFA 우회)
   - OTG-AUTHZ: 권한 부여 (IDOR·수직·수평 권한 상승)
   - OTG-SESS: 세션 관리 (세션 고정·예측 가능 토큰)
   - OTG-INPVAL: 입력 유효성 (SQLi·XSS·XML Injection·SSRF)
   - OTG-ERR: 오류 처리 (스택 트레이스 노출)
   - OTG-CRYPT: 암호화 (약한 알고리즘·키 관리 결함)
   - OTG-BUSLOGIC: 비즈니스 로직 (워크플로 우회·경쟁 조건)
   - OTG-CLIENT: 클라이언트 측 (CSRF·Clickjacking·오픈 리다이렉트)
2. **API Security Top 10 검증**: REST/GraphQL/HL7 FHIR API 에 대해 OWASP API Security Top 10 항목(API1~API10)을 적용한다.
3. **의료기기 특화 시나리오**:
   - 인증 우회를 통한 진단 데이터 조작 시나리오
   - 원격 SW 업데이트 메커니즘 무결성 우회
   - 의료 데이터 포맷(DICOM·HL7) 파서 공격

#### C. 악용 단계 (Exploitation)

1. 탐지된 취약점에 대해 계획서 승인 범위 내에서 실제 악용 가능성을 검증한다.
2. 악용 성공 시 공격 범위(데이터 접근·권한 획득·서비스 중단)를 문서화한다.
3. 악용 시도는 항상 비상 연락처 담당자에게 사전 통보 후 수행하며, 심각한 충돌·서비스 중단 발생 시 즉시 중단한다.
4. **의도하지 않은 데이터 접근**: 실수로 실제 데이터(환자 정보)에 접근한 경우 즉시 PSO·DPO 에 보고하고 데이터 처리 지침에 따라 삭제·기록한다.

#### D. 보고 단계 (Reporting)

1. **취약점 분류**: 발견 취약점을 CVSS v3.1 기준 Critical/High/Medium/Low/Informational 로 분류한다.
2. **TMP-TEST-005 작성**: 각 취약점에 대해 다음을 기재한다.
   - 취약점 제목·CVSS 점수·CWE·OWASP 분류
   - 재현 절차 (단계별, 스크린샷·요청/응답 포함)
   - 영향 범위 및 비즈니스 위험
   - 권고 조치 및 참고 자료
3. **경영진 요약**: 기술 세부 내용 외에 PSC·경영진용 1페이지 요약(발견 건수·Critical 건수·전체 위험 수준)을 포함한다.
4. **취약점 목록 통합**: TMP-RISK-006 취약점 워크시트에 펜테스트 발견 항목을 추가한다.

#### E. 재검증 단계 (Retest)

1. R&D 가 취약점 조치를 완료한 후, 최초 발견자(내부 QA 또는 외부 업체)가 동일 공격 벡터로 재검증을 수행한다.
2. 재검증 결과(조치 확인 또는 조치 불완전)를 TMP-TEST-005 에 추가 기재한다.
3. 조치 완료 확인이 된 취약점만 'Closed' 상태로 갱신한다.

### 5.3 완료 조건

- [ ] 펜테스트 계획서(TMP-TEST-005A)가 PSO·VP of R&D 의 서명 승인을 받았다
- [ ] OWASP WSTG 체크리스트의 10개 범주와 API Security Top 10 이 모두 검증되었다
- [ ] 커버리지가 ≥ 90% (주요 유스케이스 기준, PRO-MDCS-301 §7 KPI)임이 확인되었다
- [ ] 발견된 모든 취약점이 TMP-TEST-005 에 등급·재현 절차와 함께 기재되었다
- [ ] Critical·High 취약점이 이슈 트래커에 등록되었다
- [ ] Critical 취약점의 조치 또는 릴리스 차단 결정이 기록되었다
- [ ] 조치 완료 항목의 재검증이 수행되었다
- [ ] 취약점 결과가 TMP-RISK-006 에 통합되었다
- [ ] 경영진 요약 보고서가 PSC 에 제출되었다

## 6. 인터페이스 부서

- **R&D**: 환경 구성 지원, 화이트박스 정보 제공 (화이트박스 테스트 시), 취약점 조치
- **PSO**: 범위 승인, 보안 영향 판단, 잔여 위험 연계
- **QA**: 내부 펜테스트 실행, 외부 업체 조율, 결과 보고서 검토
- **외부 보안 업체**: 연 1회 이상 독립적 외부 펜테스트 수행
- **DPO**: 테스트 중 개인정보 접근 발생 시 보고·처리
- **PSC**: 결과 검토, 고위험 취약점 대응 방향 결정

## 7. 주의사항 / 예외 처리

### 7.1 외부 업체 선정 시 자격 미달 또는 보안 자격증 부재

외부 펜테스트 업체는 OSCP·CEH·CREST·GPEN 등 공인 자격증 보유자를 포함해야 하며, 의료기기·OT/IoT 보안 경험이 있는 업체를 우선 선정한다. 자격 미달 업체와의 계약은 PSO 가 거부할 수 있다. 선정 기준과 검토 결과를 계획서에 기재한다.

### 7.2 테스트 중 실제 서비스 장애 발생

테스트 과정에서 In-scope 외 시스템 또는 공유 인프라에 의도치 않은 장애가 발생한 경우, 즉시 테스트를 중단하고 비상 연락처에 통보한다. 서비스 복구를 최우선으로 하며, 장애 원인·영향 범위·복구 조치를 사고 보고서로 작성한다. 재개 전 범위 재검토를 수행한다.

### 7.3 테스트 결과 기밀 누출 우려

외부 업체가 작성한 펜테스트 리포트는 기밀 문서로 분류하여 암호화된 채널로만 전달받는다. 리포트 수신 후 외부 업체의 로컬 복사본 삭제를 계약 조건에 포함한다. 결과 데이터 처리 이력을 TMP-TEST-005A 에 기재한다.

### 7.4 시간 제약으로 OWASP 전체 커버리지 달성 불가

릴리스 일정상 전체 OWASP 커버리지가 불가한 경우, PSO 와 QA 가 STRIDE 위협 모델링 결과를 바탕으로 **고위험 공격 경로 우선** 방식으로 범위를 조정하고 PSC 에 축소 승인을 받는다. 미수행 항목은 차기 릴리스 또는 연간 외부 펜테스트에서 보완하고 일정을 계획서에 기재한다.
