# 문서 갱신 및 폐기 업무지침 (WI-101-02)

> 상위 절차: [[PRO-MDCS-101_보안_문서화_관리_절차_v1.0]] · §4 흐름 C단계·§5 단계 2~8

## 1. 업무 목적

SaMD-CSMS 제02조에 따라 승인·배포된 보안 문서를 **경미 개정** 또는 **중대 개정**으로 분류하여 적합한 심의·승인 경로를 거쳐 갱신하고, 더 이상 유효하지 않은 문서를 **폐기** 처리하여 구 버전이 현업에서 사용되지 않도록 통제한다. 폐기된 문서는 규제 추적을 위해 별도 아카이브에 이관하여 지정 기간 동안 보존한다.

## 2. 수행 주체

- **주 수행자**: Doc Owner (해당 문서 담당 부서·개인), PSO
- **경미 개정 검토·승인자**: QMR
- **중대 개정 심의·승인자**: PSC (심의), CISO (승인)
- **폐기 확인자**: PSO, QMR

## 3. 범위

- MAT-001(문서 관리 대장)에 등록된 **모든 SaMD-CSMS 보안 문서** (POL/PRO/WI/TMP/EX/MAT/REF)
- 연 1회 유효성 검토 결과 갱신이 필요하다고 판단된 문서
- 운영 환경·규제 변경·중대 사이버보안 사건으로 즉시 개정이 필요해진 문서
- 기능 통합·구조 변경으로 더 이상 독립 운용이 불필요한 문서 (폐기 대상)

## 4. 입력 자료 / 산출물

- **Input**
  - 현행 승인 문서 최신본 (MAT-001 참조)
  - 개정 요청 사유서 또는 유효성 검토 결과 보고서
  - 관련 규제·표준 변경 공지 (해당 시)
  - 연계 문서 목록 (개정 시 영향 범위 파악)
- **Output**
  - 개정본 (새 버전 번호 부여, MAT-001 업데이트)
  - 폐기 확인서 (TMP-DOC-004)
  - 폐기본 아카이브 이관 기록 (REC-DOC-{YYYY}-{###})
  - 개정/폐기 사유 및 영향 범위 요약 메모

## 5. 수행 절차

### 5.1 사전 준비

1. Doc Owner 는 개정·폐기 필요성이 식별되면 PSO 에 **개정 요청서(TMP-DOC-003)**를 제출한다.
2. PSO 는 요청서를 검토하여 **경미/중대** 분류 기준을 적용한다.
   - **경미**: 오탈자·링크 수정, 담당자명 변경, 서식 조정, 비핵심 설명 보완
   - **중대**: 절차 흐름·RACI 변경, 적용 범위 확장/축소, 통제 요구사항 수준 변경, 규제 요건 반영, 새로운 섹션 추가·삭제
3. 개정 범위에 따라 영향받는 연계 문서(상·하위 및 참조 문서) 목록을 작성한다.
4. 폐기의 경우 후속 문서(대체 문서) 또는 기능 흡수 문서를 명시한다.

### 5.2 수행 단계

#### A. 경미 개정 처리

1. **초안 작성**: Doc Owner 가 현행 문서에서 수정 사항을 반영하여 초안 작성. 버전을 마이너(예: 1.0 → 1.1)로 올리고, frontmatter 의 `updated` 날짜를 갱신한다.
2. **QMR 검토**: QMR 이 경미 개정 적합성과 연계 문서 일관성을 검토. 검토 결과를 **개정 요청서(TMP-DOC-003)** 하단 검토란에 기재한다.
3. **QMR 승인**: 검토 이상 없을 시 QMR 이 승인 서명. 별도 PSC 심의 불필요.
4. **배포 및 등록**: Doc Owner 가 MAT-001 버전 정보를 업데이트하고 구 버전을 `[SUPERSEDED]` 표시. 이해관계자에게 변경 공지.

#### B. 중대 개정 처리

1. **초안 작성**: Doc Owner 가 초안을 작성. 버전을 메이저(예: 1.x → 2.0)로 올리고 `updated` 날짜를 갱신한다.
2. **PSC 심의**: PSO 가 PSC 회의 안건으로 상정. PSC 는 개정 타당성·위험 통제 적합성·규제 부합 여부를 심의하고 결과를 의사록에 기재.
3. **CISO 승인**: PSC 심의 결과를 첨부하여 CISO 최종 승인. 중대 개정은 반드시 CISO 서명이 필요하다.
4. **영향 문서 연쇄 갱신**: 영향을 받는 연계 문서(하위 WI, TMP, MAT 등)를 순차 갱신하거나 개정 요청을 발행한다.
5. **배포 및 등록**: MAT-001 업데이트 후 전 이해관계자에게 공지. 구 버전 `[SUPERSEDED]` 표시.

#### C. 폐기 처리

1. **폐기 요청 검토**: PSO 가 폐기 대상 문서의 사용 현황, 연계 참조, 규제 의무를 확인한다.
2. **대체 문서 지정**: 폐기 문서의 기능을 흡수하는 대체 문서(또는 "폐지 후 불필요"임을 명시)를 지정한다.
3. **PSC 심의**: 폐기 결정도 중대 변경으로 간주하여 PSC 심의를 거친다.
4. **폐기 확인서 작성**: TMP-DOC-004 에 폐기 사유, 대체 문서, 승인자, 보존 기간(최소 10년)을 기재한다.
5. **아카이브 이관**: 폐기본을 `08_REC_기록/_archive/` 폴더로 이관하고 문서명 앞에 `[OBSOLETE-YYYY-MM-DD]` 접두어를 붙인다.
6. **MAT-001 상태 갱신**: 해당 문서 행의 상태를 `obsolete`로 변경하고 폐기일자·아카이브 경로를 기재한다.
7. **현업 버전 제거**: 현업 접근 폴더(vault/05_WI 등)에서 파일을 삭제하거나 읽기 전용으로 잠그고, 관련자에게 폐기 통보 발송.

### 5.3 완료 조건

- [ ] 경미 개정의 경우 QMR 검토·승인 서명이 개정 요청서(TMP-DOC-003)에 존재한다
- [ ] 중대 개정의 경우 PSC 심의 의사록과 CISO 승인 서명이 존재한다
- [ ] MAT-001 에 새 버전 정보(버전번호·승인일·담당자)가 등록되었다
- [ ] 구 버전 문서가 `[SUPERSEDED]` 표시되거나 아카이브로 이관되었다
- [ ] 폐기의 경우 폐기 확인서(TMP-DOC-004)가 작성·승인되었다
- [ ] 폐기 문서가 `_archive/` 에 `[OBSOLETE-YYYY-MM-DD]` 접두어로 이관되었다
- [ ] 연계 문서의 개정 필요 여부가 검토되었고 필요 시 개정 요청이 발행되었다
- [ ] 이해관계자에게 변경/폐기 공지가 발송되었다

## 6. 인터페이스 부서

- **Doc Owner (개발·SecOps·CSIRT 등)**: 개정 초안 작성, 폐기 요청 제출
- **QMR**: 경미 개정 검토·승인, 문서 관리 시스템(MAT-001) 관리
- **PSC**: 중대 개정·폐기 심의
- **CISO**: 중대 개정 최종 승인
- **IT/시스템 관리**: vault 파일 시스템 권한 제어, 아카이브 폴더 관리

## 7. 주의사항 / 예외 처리

### 7.1 규제 기관 요청으로 즉시 개정이 필요한 경우

식품의약품안전처 등 규제 기관이 특정 문서의 즉시 개정을 요구하는 경우, PSO 는 72시간 이내에 초안을 완성하고 PSC 긴급 서면 심의(이메일 회람 가능)를 통해 CISO 임시 승인을 받는다. 정식 서명은 72시간 이내 후속 완료한다. 긴급 처리 사실과 사유를 MAT-001 비고란에 기재한다.

### 7.2 폐기 대상 문서가 현재 진행 중인 심사(심사·인증·감사)에 참조된 경우

심사가 완료되기 전에는 폐기 절차를 일시 보류하고, 해당 문서를 `[PENDING-OBSOLETE]` 상태로 마킹한다. 심사 완료 후 즉시 폐기 절차를 재개한다. 심사 담당자(QMR)에게 보류 사실을 통보하고 심사 일정과 연동하여 관리한다.

### 7.3 연쇄 개정 범위가 3건을 초과하는 경우

단일 개정이 3개 이상의 연계 문서 개정을 유발하는 경우, PSO 는 전체 영향 범위를 분석한 **연쇄 개정 계획서**를 작성하고 PSC 에 보고한다. 개정 일정·담당·우선순위를 계획서에 명시하여 순차적으로 처리한다.

### 7.4 버전 번호 불일치 발견

이미 배포된 문서의 버전 번호가 MAT-001 기록과 불일치하는 경우, PSO 가 즉시 원인을 조사하고 정확한 버전을 MAT-001 에 정정 기재한 후 정정 사유를 이력 메모로 남긴다. 의도하지 않은 미승인 변경이 확인되면 PSC 에 보안 사고로 보고한다.
