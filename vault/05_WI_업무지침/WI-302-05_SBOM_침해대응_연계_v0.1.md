# SBOM 침해대응 연계 업무지침 (WI-302-05)

> 상위 절차: [[PRO-MDCS-302_SBOM_관리_절차_v1.0]] · §5 단계 8

## 1. 업무 목적

침해사고 발생 시 공격자가 악용한 소프트웨어 구성요소를 신속히 식별하고 동일 컴포넌트를 포함한 영향 제품군을 전수 조회하기 위해, SBOM 데이터베이스를 침해사고 대응(IR) 프로세스에 연계하는 절차를 정의한다. 본 지침은 CSIRT가 공격 벡터 분석 및 영향 범위 확정에 SBOM을 활용하고, 결과를 [[PRO-MDCS-502_침해사고_대응_절차]] 에 인계하는 인터페이스를 규정한다.

## 2. 수행 주체

- **주 수행자**: SBOM 관리자 (SBOM 조회·영향 분석·보고서 작성)
- **요청자**: CSIRT 리더 (침해사고 대응 담당, 조회 요청)
- **검토자**: PSO (분석 결과 검토·영향 등급 결정)
- **승인자**: CISO (영향 범위 확정·외부 통보 승인)
- **협력**: R&D (구성요소 기술 세부사항), 고객지원팀 (MSP 통보 지원)

## 3. 범위

- 침해사고 탐지 후 CSIRT가 개시하는 영향 분석 단계에서의 SBOM 활용
- 공격에 사용된 취약점 CVE와 SBOM 구성요소의 교차 분석
- 영향 받는 제품·버전·컴포넌트 목록 생성 및 우선순위 지정
- 분석 결과를 PRO-MDCS-502 IR 프로세스의 봉쇄·근절 단계에 인계

## 4. 입력 자료 / 산출물

- **Input**
  - CSIRT의 침해사고 티켓 (사고 ID·탐지 일시·초기 공격 벡터 정보)
  - 공격에서 확인된 또는 의심되는 CVE ID 또는 취약 구성요소명·버전
  - SBOM 저장소 내 모든 릴리스별 SBOM 파일
  - CVE/NVD 연동 데이터 (WI-302-02 산출물)
- **Output**
  - SBOM 기반 영향 분석 보고서 (영향 제품·버전·컴포넌트 목록, 배포 현황 포함)
  - 우선 조치 대상 목록 (Critical/High CVE 및 의료 기능과 연계된 컴포넌트 우선)
  - PRO-MDCS-502 봉쇄·근절 단계 인계 패키지 (분석 보고서 + 조치 권고 사항)
  - MSP 통보 대상 목록 (영향 제품을 운용 중인 MSP 현황)

## 5. 수행 절차

### 5.1 사전 준비

1. SBOM 관리자는 침해 대응 시나리오에서 SBOM 조회가 즉시 가능하도록 **오프라인 SBOM 스냅샷**(최신 릴리스 기준 분기 1회)을 보안 격리된 스토리지에 유지한다. 네트워크 침해로 메인 저장소 접근이 불가능한 경우 이 스냅샷을 사용한다.
2. SBOM 검색 도구(예: SPDX Tools, CycloneDX CLI 또는 자체 스크립트)를 사전 검증하고, 구성요소명·버전·CPE·PURL 기반 교차 검색이 정상 동작하는지 분기 1회 테스트한다.
3. CSIRT와 연락처 매트릭스를 최신 상태로 유지하고, 침해 대응 개시 시 자동 알림을 수신하도록 ITSM 티켓 시스템을 구성한다.

### 5.2 수행 단계

1. **침해 대응 개시 통보 수신**
   - CSIRT 리더가 PRO-MDCS-502의 탐지·분류 단계를 완료하고 SBOM 분석이 필요하다고 판단하면, SBOM 관리자에게 공식 조회 요청을 전달한다. 요청에는 사고 ID, 관련 CVE 또는 의심 구성요소 정보가 포함되어야 한다.
   - SBOM 관리자는 요청 수신을 확인하고 분석 착수 시각을 기록한다. 목표 완료 시간: Critical 사고는 수신 후 **2시간**, High 사고는 **4시간**, 그 외는 **24시간** 이내.

2. **CVE·구성요소 기반 SBOM 교차 검색**
   - SBOM 검색 도구를 사용하여 SBOM 저장소 내 모든 릴리스를 대상으로 다음 기준으로 검색한다.
     - 지정된 CVE ID와 매칭되는 구성요소(CPE 또는 PURL 기반)
     - 취약 버전 범위에 포함되는 구성요소 버전
     - 의심 구성요소명이 포함된 모든 제품·버전
   - 검색 결과로 영향 제품 목록(제품명·릴리스 버전·구성요소명·버전·의존 유형(직접/전이))을 생성한다.

3. **영향 등급 분류 및 우선순위 지정**
   - PSO와 협력하여 각 영향 구성요소에 대해 의료 기능 연관성과 CVE CVSS 점수를 결합한 위험 등급을 산정한다.
   - 환자 안전·핵심 의료 기능(진단·치료 관련)에 직접 연계된 구성요소는 CVSS 점수와 무관하게 최우선 조치 대상으로 분류한다.
   - 위험 등급별 조치 권고 사항(긴급 패치·업데이트·운용 제한·모니터링 강화)을 작성한다.

4. **영향 분석 보고서 작성**
   - 다음 항목을 포함하는 보고서를 작성한다.
     - 사고 개요 및 분석 기준 (CVE ID·의심 구성요소)
     - 영향 제품·버전 전체 목록 (배포 현황: 내부 테스트/MSP 납품/현장 운용)
     - 우선 조치 대상 목록
     - 영향 없음 확인 제품·버전 목록 (범위 제한 근거 포함)
     - 분석 한계 사항 (SBOM 미생성 릴리스, 데이터 불완전 등)

5. **PRO-MDCS-502 봉쇄·근절 단계 인계**
   - 분석 보고서를 CSIRT 리더에게 전달하고, JIRA 사고 티켓에 첨부하여 공식 기록으로 남긴다.
   - CSIRT는 보고서를 기반으로 영향 시스템 격리·패치·업데이트 조치를 실행한다.
   - SBOM 관리자는 조치 완료 후 해당 구성요소가 수정된 버전으로 갱신된 새 SBOM이 생성되는지 R&D와 확인한다.

6. **MSP 통보 지원**
   - 현장에 배포된 제품이 영향을 받는 경우, 고객지원팀과 협력하여 영향 MSP 목록과 긴급 통보 내용(영향 제품·버전·권고 조치)을 준비한다.
   - MSP 통보는 PRO-MDCS-502 외부 통보 단계 및 WI-302-04 MSP 제공 채널을 통해 진행한다.

### 5.3 완료 조건

- [ ] CSIRT 요청 수신 후 목표 시간(Critical 2h / High 4h / 기타 24h) 내에 분석 보고서가 작성되었다
- [ ] 영향 제품·버전 목록과 영향 없음 확인 목록이 모두 보고서에 포함되어 있다
- [ ] PSO와 협력하여 우선 조치 대상 목록에 의료 기능 연관성 등급이 반영되었다
- [ ] 보고서가 PRO-MDCS-502 사고 티켓에 공식 첨부되어 인계 완료되었다
- [ ] 현장 배포 제품 영향 시 MSP 통보 대상 목록이 고객지원팀에 전달되었다
- [ ] 조치 후 갱신된 SBOM 생성 여부가 R&D와 확인되었다

## 6. 인터페이스 부서

- **CSIRT**: 침해사고 대응 개시·진행·종결, SBOM 분석 결과 수신 및 봉쇄·근절 조치
- **PSO**: 영향 등급 결정, 의료 기능 연관성 평가
- **R&D**: 취약 구성요소 기술 세부사항 제공, 갱신 SBOM 생성
- **고객지원팀**: MSP 통보 이메일 발송 및 수신 확인 관리
- **CISO**: 영향 범위 및 외부 통보 범위 최종 승인

## 7. 주의사항 / 예외 처리

### 7.1 네트워크 침해로 SBOM 저장소 접근 불가

네트워크 침해로 인해 메인 SBOM 저장소에 접근이 불가능한 경우, §5.1에서 유지하는 격리 오프라인 스냅샷을 사용하여 분석을 진행한다. 스냅샷의 최신화 날짜와 메인 저장소 최신 릴리스 날짜 사이의 갭을 보고서에 명시하고, 갭 기간의 신규 릴리스에 대해서는 R&D에서 빌드 아티팩트 정보를 수동으로 제공받아 보완한다.

### 7.2 SBOM 미생성 릴리스

분석 대상 제품 중 SBOM이 생성되지 않은 릴리스(레거시 버전)가 있는 경우, 해당 사실을 보고서의 분석 한계 섹션에 명시하고 R&D에 해당 릴리스의 의존성 목록을 수동으로 요청한다. 수동 의존성 목록이 제공된 경우 임시 SBOM 초안을 생성하여 분석에 사용하고, 공식 SBOM 생성을 R&D에 요청한다.

### 7.3 공격 벡터가 불명확한 초기 단계 조회

CVE 또는 구성요소 정보가 특정되지 않은 상태에서 조기 분석이 요청되는 경우, SBOM 관리자는 침해 징후(관련 로그 패턴·파일명·네트워크 연결 정보)를 기반으로 가능성 있는 구성요소 후보를 PSO와 협의하여 1차 범위를 설정하고 분석을 시작한다. 추가 포렌식 정보가 수신될 때마다 분석을 갱신하고 보고서를 버전 관리한다.

- 오프라인 스냅샷: 보안 격리 스토리지 (분기 갱신 로그 포함)
