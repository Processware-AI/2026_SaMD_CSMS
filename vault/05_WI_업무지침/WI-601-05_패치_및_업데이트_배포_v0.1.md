# 패치 및 업데이트 배포 업무지침 (WI-601-05)

> 상위 절차: [[PRO-MDCS-601_취약점_감시_공개_조치_절차_v1.0]] · §5 단계 6~8

## 1. 업무 목적

취약점 조치·유지관리 목적의 패치를 제10조 유지관리 기준에 부합하는 방식으로 개발·서명·배포하고, 배포 실패 시 롤백과 복구가 보장되도록 한다.

## 2. 수행 주체

- **주 수행자**: 개발팀, 인프라팀
- **검토자**: QA 리드, PSO, SBOM 관리자
- **승인자**: 오동석 (릴리스 게이트)

## 3. 범위

- 애플리케이션 패치·펌웨어 업데이트
- OS·제3자·오픈소스 보안 업데이트
- 구성 변경 기반 완화(Workaround)

## 4. 입력 자료 / 산출물

- **Input**
  - 취약점 티켓·영향 분석
  - 패치 요구사항·승인된 SLA
- **Output**
  - 서명된 패치 빌드 + SHA-256 해시
  - [[TMP-VUL-004_패치_배포_체크리스트_v0.1]]
  - 배포 이력·롤백 기록

## 5. 수행 절차

### 5.1 사전 준비

1. 개발팀이 릴리스 파이프라인에 자동 서명·해시 생성 단계를 구성한다.
2. 공식 배포 채널(OTA·포털·파트너 포털) 을 HTTPS/TLS 로 운영한다.
3. 롤백 전략을 사전에 설계(이전 버전 보관·구성 스냅샷).

### 5.2 수행 단계

1. **개발**
   - 보안 SDLC([[PRO-MDCS-301_보안_SDLC_절차_v1.0]]) 에 따라 패치 개발.
   - 변경 범위를 최소화하고 영향 분석 완료.
2. **QA 보안 테스트**
   - 회귀·보안 테스트 수행. SAST/DAST 통과 요건 충족.
   - 재현 환경에서 취약점 완전 해소 검증.
3. **서명·해시**
   - 코드 서명 인증서로 패치 서명.
   - SHA-256 해시를 동반 제공.
4. **단계적 배포**
   - 카나리(1%) → 파일럿 MSP → 전체.
   - 각 단계에서 건강도 지표(실패율·장애 알림) 를 모니터링.
5. **롤백**
   - 실패율이 임계(예: 2%) 초과 시 자동 롤백 트리거.
   - 수동 롤백 절차는 오동석 승인 후 수행.
6. **배포 완료·기록**
   - [[TMP-VUL-004_패치_배포_체크리스트_v0.1]] 완성·승인.
   - SBOM 갱신 및 CVE 연동 재실행.

### 5.3 완료 조건

- [ ] 패치가 보안 SDLC 단계를 모두 통과했다
- [ ] QA 보안 테스트에서 취약점 해소·회귀 없음이 확인되었다
- [ ] 디지털 서명·SHA-256 해시가 제공된다
- [ ] 카나리·파일럿·전체의 단계적 배포가 수행되었다
- [ ] 롤백 자동 트리거와 수동 절차가 작동 가능하다
- [ ] 배포 이력·SBOM 갱신이 기록되었다

## 6. 인터페이스 부서

- **SBOM 관리자**: 갱신된 SBOM 배포
- **PSO**: 게이트 리뷰
- **홍보/CS**: 사용자 공지
- **RA**: 규제 통보와 패치 배포 동기화

## 7. 주의사항 / 예외 처리

### 7.1 긴급 핫픽스

Critical 취약점의 7일 SLA 를 충족하기 위해 단계적 배포를 건너뛰어야 하는 경우, 오동석 승인 아래 전체 즉시 배포로 전환한다. 사후 72시간 집중 모니터링·롤백 준비를 강화한다.

### 7.2 디지털 서명 키 유출

코드 서명 키 유출 의심 시 즉시 해당 인증서 폐기·재발급, 이미 배포된 패치의 재서명·재배포 결정을 오동석 가 내린다. 취약 서명 패치 제거 지침을 사용자에 공지.

### 7.3 MSP 환경 호환성 충돌

배포 후 특정 MSP 환경에서 호환성 문제 발생 시, 대상 MSP 에만 선택적 롤백 또는 우회 구성 배포를 수행한다. 근본 원인 분석 후 정식 재배포.

### 7.4 네트워크 단절 기기 (오프라인 의료기기)

네트워크 단절 상태 기기는 수동 업데이트 키트(서명된 USB) 로 배포한다. 수령·적용 확인을 MSP 서명으로 받고 기록 보존.
