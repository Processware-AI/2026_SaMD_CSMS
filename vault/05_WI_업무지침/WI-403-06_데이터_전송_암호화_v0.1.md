# 데이터 전송 암호화 업무지침 (WI-403-06)

> 상위 절차: [[PRO-MDCS-403_데이터_및_파일_보안_관리_절차_v1.0]] · §5 단계 7

## 1. 업무 목적

시스템 간·클라이언트-서버 간·내부 서비스 간 전송되는 모든 데이터를 TLS 1.2 이상으로 암호화하고, 민감 데이터(개인정보·의료정보)에는 전송 계층 암호화 외 자체(애플리케이션 레벨) 암호화를 추가 적용(이중 보안)하여, 네트워크 도청·중간자 공격(MITM)·전송 중 데이터 탈취를 예방한다. SaMD-CSMS 제08조 제3호(전송 암호화·이중 보안, MDCS-R-083) 및 제08조 제4호(표준 알고리즘·취약 알고리즘 금지, MDCS-R-084) 요건을 충족한다.

## 2. 수행 주체

- **주 수행자**: 개발팀 보안담당 (TLS 설정, 인증서 발급 신청, 애플리케이션 암호화 구현)
- **공동 수행자**: 인프라팀 (로드밸런서·API 게이트웨이 TLS 종단 설정, 인증서 갱신 자동화)
- **검토자**: SecOps (취약 프로토콜·cipher suite 탐지, 인증서 만료 모니터링)
- **승인자**: CISO (표준 미달 예외 허용 시)

## 3. 범위

- **외부 통신**: 의료기기-클라우드 서버, 클라이언트 웹/앱-API 서버, 외부 파트너 시스템 간 API
- **내부 서비스 간 통신**: 마이크로서비스·컨테이너 간 gRPC/REST, DB 클라이언트-서버, 메시지 큐(Kafka, RabbitMQ), 내부 API 호출
- **데이터 분류별 추가 적용**: 개인정보·의료정보·인증정보 전송 시 애플리케이션 레벨 자체 암호화 필수
- **제외**: 동일 서버 내 IPC(루프백 인터페이스 127.0.0.1) — 단, 컨테이너 간 통신은 포함

## 4. 입력 자료 / 산출물

- **Input**
  - 서비스 아키텍처 다이어그램 (통신 경로 식별용)
  - [[TMP-TLS-002_전송_암호화_적용_현황표_v0.1]] (신규 서비스 추가 또는 변경 시 업데이트)
  - CA 인증서 발급 요청 정보 (FQDN·SAN·유효기간)
- **Output**
  - [[TMP-TLS-001_인증서_관리_대장_v0.1]] (인증서 ID·FQDN·발급 CA·유효기간·갱신 담당자·자동 갱신 여부)
  - [[TMP-TLS-002_전송_암호화_적용_현황표_v0.1]] (서비스별 TLS 버전·cipher suite·자체 암호화 적용 여부)
  - TLS 스캐닝 리포트 (Qualys SSL Labs, testssl.sh 등 분기 결과)
  - 인증서 갱신 이력

## 5. 수행 절차

### 5.1 사전 준비

1. 개발팀 보안담당은 서비스 아키텍처 다이어그램을 기반으로 모든 내·외부 통신 경로를 식별하고 [[TMP-TLS-002_전송_암호화_적용_현황표_v0.1]] 를 작성한다. 신규 서비스 추가 또는 아키텍처 변경 시 1주 이내 업데이트한다.
2. 조직의 허용 TLS 설정 기준을 수립하고 개발·인프라팀에 공유한다:
   - **허용 프로토콜**: TLS 1.2, TLS 1.3
   - **금지 프로토콜**: SSL 2.0/3.0, TLS 1.0, TLS 1.1
   - **허용 Cipher Suite (TLS 1.2)**: `TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256`, `TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384`, `TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256`, `TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384`
   - **금지 Cipher**: RC4, DES, 3DES, EXPORT 등급, NULL cipher, 익명(Anonymous) DH
3. 인증서 자동 갱신(ACME/Let's Encrypt 또는 내부 CA 자동화)이 설정된 경우 갱신 성공 여부를 월 1회 확인하며, 만료 30일 전 이메일·SIEM 알림이 발송되는지 검증한다.

### 5.2 수행 단계

1. **외부 서비스 TLS 설정**
   - 로드밸런서·API 게이트웨이·웹 서버(Nginx, Apache 등)의 TLS 설정에서 TLS 1.0/1.1 비활성화 및 금지 cipher suite 제거를 적용한다.
   - HTTPS로의 강제 리다이렉트(HTTP→HTTPS) 및 HSTS(Strict-Transport-Security: max-age≥31536000; includeSubDomains) 헤더를 활성화한다.
   - 공개 서비스의 TLS 설정이 Qualys SSL Labs 등급 A 이상을 유지하도록 분기 1회 스캔한다.
2. **내부 서비스 간 TLS 적용**
   - 마이크로서비스 간 통신에 mTLS(상호 인증 TLS)를 적용하며, 서비스 메시(Istio, Linkerd 등) 또는 애플리케이션 레벨 mTLS 중 하나를 선택한다.
   - DB 클라이언트-서버 연결에 TLS를 활성화하고, 서버 인증서 검증(SSL_VERIFY_PEER)을 강제한다.
   - 메시지 큐 브로커(Kafka, RabbitMQ 등) 연결에 TLS와 클라이언트 인증서 기반 인증을 적용한다.
3. **민감 데이터 자체 암호화 (이중 보안)**
   - 개인정보·의료정보·인증정보를 전송하는 API 응답·요청 페이로드에, TLS 적용과 별도로 필드 레벨 암호화(AES-256-GCM)를 추가 적용한다.
   - 암호화 키는 [[WI-402-05_HSM_KMS_운영_v0.1]] 에 따라 KMS에서 관리한다.
   - 자체 암호화 적용 대상 API를 [[TMP-TLS-002_전송_암호화_적용_현황표_v0.1]] 에 명시하고 분기 검토 시 최신화한다.
4. **인증서 발급 및 관리**
   - 신규 서비스 도메인에 대한 TLS 인증서를 발급 전, 개발팀 보안담당이 CA 발급 요청 정보(FQDN·SAN·유효기간·서비스 소유자)를 [[TMP-TLS-001_인증서_관리_대장_v0.1]] 에 등록한다.
   - 와일드카드 인증서(`*.domain.com`)는 최소화하며, 사용 시 적용 서비스 목록을 대장에 명시한다.
   - 자체 서명(Self-signed) 인증서는 내부 테스트 환경에만 허용하며, 운영 환경에서는 신뢰할 수 있는 CA 발급 인증서만 사용한다.
   - 인증서 유효기간은 최대 397일(13개월)로 제한하며, 자동 갱신을 기본 정책으로 적용한다.
5. **인증서 갱신 및 교체**
   - 자동 갱신 실패 또는 인증서 침해 의심 시, 인프라팀이 즉시 수동 갱신을 수행하고 [[TMP-TLS-001_인증서_관리_대장_v0.1]] 갱신 이력을 기재한다.
   - 인증서 교체 시 구 인증서는 교체 완료 확인 후 폐기(Revoke)하며, 폐기 완료를 CA에서 확인한다.
6. **정기 취약점 스캔 및 검토**
   - SecOps가 분기 1회 testssl.sh 또는 Qualys SSL Labs를 사용하여 취약 프로토콜·cipher suite·인증서 문제(자체 서명, 만료 임박, 약한 키 길이)를 스캔하고 결과를 보고한다.
   - 취약 항목 발견 시 발견 후 30일 이내 조치 계획을 수립하고, Critical 취약점은 7일 이내 긴급 조치한다.

### 5.3 완료 조건

- [ ] 모든 외부 통신 엔드포인트에 TLS 1.2 이상이 적용되고 금지 프로토콜·cipher가 비활성화되었다
- [ ] 내부 서비스 간 통신에 mTLS 또는 TLS가 적용되었다
- [ ] 개인정보·의료정보 전송 API에 애플리케이션 레벨 자체 암호화(이중 보안)가 적용되었다
- [ ] 모든 인증서가 [[TMP-TLS-001_인증서_관리_대장_v0.1]] 에 등록되고 만료 30일 전 알림이 설정되었다
- [ ] [[TMP-TLS-002_전송_암호화_적용_현황표_v0.1]] 가 현행 아키텍처와 일치한다
- [ ] 분기 TLS 스캔에서 Critical 취약점 0건이 확인되었다

## 6. 인터페이스 부서

- **인프라팀**: 로드밸런서·게이트웨이 TLS 설정, 인증서 자동 갱신 인프라 운영
- **SecOps**: 취약 프로토콜 탐지, 인증서 만료 모니터링, 분기 스캔 수행
- **CISO**: 표준 미달 설정 예외 허용 승인
- **DPO**: 개인정보 전송 경로 식별 및 이중 암호화 적용 요건 정의
- **개발팀**: 애플리케이션 레벨 암호화 구현, TLS 클라이언트 검증 설정
- **외부 CA / CSP**: 인증서 발급, 클라우드 관리형 인증서 서비스(ACM, App Service Certificate 등)

## 7. 주의사항 / 예외 처리

### 7.1 TLS 인증서 긴급 교체 (침해·오발급)

인증서 개인키 노출이 의심되거나 CA에서 오발급 알림을 수신한 경우, 개발팀 보안담당과 인프라팀이 즉시(24시간 이내) 신규 키 쌍 생성 및 새 인증서 발급을 수행한다. 구 인증서를 CA에 즉시 폐기 요청(OCSP Revocation)하고, 서비스 재시작 없이 인증서 핫 리로드(hot reload)가 가능한 경우 활용한다. 사고 경위와 조치 내역을 CISO에 보고한다.

### 7.2 레거시 클라이언트의 TLS 1.0/1.1 호환성 요구

의료기기·레거시 시스템이 TLS 1.2를 지원하지 않아 TLS 1.0/1.1 유지가 불가피한 경우, CISO 예외 승인을 받고 해당 엔드포인트를 전용 격리 구간(isolated zone)으로 분리한다. 격리 구간에서는 IP 화이트리스트·추가 네트워크 통제를 적용하며, 레거시 기기 교체 또는 업그레이드 계획을 6개월 이내 수립한다. 예외는 연 1회 재검토하여 해소 여부를 확인한다.

### 7.3 클라우드 서비스 간 통신

CSP(Cloud Service Provider) 내부 네트워크를 통한 서비스 간 통신도 TLS 적용을 원칙으로 한다. CSP 프라이빗 네트워크를 신뢰하여 평문 내부 통신을 허용하는 경우, 해당 통신 경로를 [[TMP-TLS-002_전송_암호화_적용_현황표_v0.1]] 에 예외로 명시하고 CISO 승인을 받는다. 민감 데이터를 전송하는 경우에는 예외 없이 자체 암호화를 적용한다.

### 7.4 테스트·개발 환경의 자체 서명 인증서 관리

개발·테스트 환경에서 자체 서명 인증서를 사용하는 경우, 해당 인증서의 루트 CA를 시스템 신뢰 저장소에 추가하지 않고 환경별 신뢰 저장소로 격리 관리한다. 개발 환경 인증서가 운영 환경에 혼입되지 않도록 배포 파이프라인에서 인증서 Thumbprint 검증 단계를 포함한다.
