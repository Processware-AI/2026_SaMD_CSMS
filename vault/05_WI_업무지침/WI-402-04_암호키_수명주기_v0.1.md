# 암호키 수명주기 관리 업무지침 (WI-402-04)

> 상위 절차: [[PRO-MDCS-402_보안_통신_및_암호화_관리_절차_v1.0]] · §5 단계 6~8

## 1. 업무 목적

디지털의료기기와 운영 환경에서 사용되는 모든 대칭·비대칭 암호화 키의 생성부터 폐기까지 전 생애주기를 HSM/KMS 기반으로 통제하고, 키 접근을 최소권한·MFA 로 강화하여 키 노출·탈취를 예방한다.

## 2. 수행 주체

- **주 수행자**: KMS Admin
- **검토자**: 인프라팀, CISO
- **승인자**: CISO

## 3. 범위

- 데이터 암호화 키(DEK) 및 키 암호화 키(KEK)
- TLS·VPN 개인키, 코드 서명 키, JWT 서명 키
- 기기 내장 키(TPM/SE 보호)
- KMS 루트 키·HSM 마스터 키

## 4. 입력 자료 / 산출물

- **Input**
  - 키 용도 목록(애플리케이션별)
  - [[TMP-NET-003_암호키_인벤토리_v0.1]] 양식
- **Output**
  - 키 인벤토리(용도·알고리즘·생성일·로테이션 주기·폐기일)
  - 키 로테이션 로그
  - 접근 로그 및 감사 리포트

## 5. 수행 절차

### 5.1 사전 준비

1. KMS Admin 이 키 사용 용도 카탈로그를 유지한다. 신규 용도는 사전 승인 후 키 생성.
2. 모든 키는 원칙적으로 HSM/KMS 내부에서 생성·저장·사용한다.
3. 키 사용 정책(용도별 유효기간·로테이션 주기) 을 확정한다: 대칭 DEK 1년, TLS 1년, 코드 서명 3년(HSM 내부).

### 5.2 수행 단계

1. **키 생성**
   - HSM/KMS 내부에서 FIPS 140-3 Level 2 이상 모듈로 생성한다.
   - 개인키는 외부로 내보내지 않는다(non-exportable).
   - 생성 즉시 [[TMP-NET-003_암호키_인벤토리_v0.1]] 에 등록한다.
2. **키 배포·사용**
   - 애플리케이션은 KMS API 호출 또는 HSM Wrap/Unwrap 으로만 사용한다.
   - 키 평문 값은 어떤 파일·로그에도 기록하지 않는다.
3. **로테이션**
   - 용도별 주기에 따라 자동 로테이션을 구성한다.
   - 로테이션 시 구 버전은 복호화 전용으로만 유지하고, 30일 이후 폐기한다.
   - 로테이션 실패 시 SIEM 경보를 발생시키고 KMS Admin 이 24시간 이내 조치한다.
4. **백업·복구**
   - KEK·루트키는 N-of-M 스킴(예: 3-of-5) 으로 분할 백업한다.
   - 백업본은 물리적으로 분리된 금고에 보관하고, 반기 1회 복구 모의훈련을 수행한다.
5. **접근 통제**
   - KMS 콘솔·HSM 관리 인터페이스는 MFA 필수.
   - 이중 통제(2인 승인) 가 요구되는 작업(루트키 사용·백업 복원) 을 정책으로 강제한다.
6. **폐기**
   - 용도 종료 시 HSM 내부에서 영구 삭제(Zeroize) 한다.
   - 폐기 전 해당 키로 보호된 데이터의 재암호화를 완료한다.
   - 폐기 기록을 인벤토리에 남긴다.

### 5.3 완료 조건

- [ ] 모든 운영 키가 HSM/KMS 내부에 저장된다
- [ ] 키 인벤토리가 모든 활성 키를 포함하며 최신 상태이다
- [ ] 용도별 로테이션 주기가 설정·자동 실행된다
- [ ] 루트키 백업은 N-of-M 스킴으로 분리 보관된다
- [ ] KMS·HSM 접근은 MFA + 2인 승인(특권 작업) 으로 강제된다
- [ ] 폐기된 키는 Zeroize 기록과 함께 인벤토리에 반영된다

## 6. 인터페이스 부서

- **인프라팀**: KMS/HSM 배포·운영 지원
- **개발팀**: 애플리케이션 KMS API 연동
- **보안감사**: 분기 접근 로그·2인 승인 증적 감사
- **SecOps**: KMS 이상 탐지 SIEM 연동

## 7. 주의사항 / 예외 처리

### 7.1 키 유출 의심

KMS Admin 이 키 유출 의심(비정상 API 호출·평문 노출 로그 등) 을 인지한 경우 즉시 해당 키를 비활성화하고 신규 키로 재암호화 계획을 수립한다. [[PRO-MDCS-502_침해사고_대응_및_신고_절차_v1.0]] 의 IR 프로세스에 에스컬레이션한다.

### 7.2 HSM 장애

HSM 장애로 서비스 중단 위험이 있는 경우, HA 구성의 대기 HSM 으로 자동 전환한다. 전환 실패 시 CISO 승인 아래 임시 소프트웨어 KMS(PKCS#11 소프트토큰) 로 단기 운용하며, 48시간 이내 HSM 복구한다.

### 7.3 알고리즘 퇴역 (PQC 전환)

NIST·KISA 에서 기존 알고리즘 취약성 공지 시 KMS Admin 이 교체 로드맵을 수립한다. 기존 키는 하이브리드 암호(기존+PQC) 로 전환 배포한 뒤 완전 교체한다.

### 7.4 외부 위탁 CA·KMS 계약 해지

클라우드 KMS·외부 CA 서비스 계약 해지 시, KMS Admin 은 키 내보내기(가능한 경우) 또는 재발급 기반 마이그레이션 계획을 세우고 데이터 재암호화를 계약 만료 60일 이전에 완료한다.
