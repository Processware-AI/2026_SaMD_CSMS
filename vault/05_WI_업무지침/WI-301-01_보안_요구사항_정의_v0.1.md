# 보안 요구사항 정의 업무지침 (WI-201-01)

> 상위 절차: [[PRO-MDCS-301_보안_SDLC_절차_v1.0]] · §5 단계 1

## 1. 업무 목적

SaMD 제품의 요구 단계에서 Security by Design 원칙에 따라 보안 요구사항을 식별·명세하여 이후 설계·구현·검증 단계에서 일관되게 추적·검증되도록 한다.

## 2. 수행 주체

- **주 수행자**: PM, R&D 리드
- **검토자**: PSO, 임상 담당자
- **승인자**: VP of R&D

## 3. 범위

- 신규 제품의 초기 요구 정의
- 주요 기능 개정 릴리스의 추가 요구
- 규제 변경에 따른 보안 요구 갱신

## 4. 입력 자료 / 산출물

- **Input**
  - 제품 기획서·사용자 유스케이스
  - SaMD-CSMS 제3~22조 요구사항 분해표
  - 임상 위험 식별 결과
- **Output**
  - [[TMP-SDLC-002_보안_요구사항_정의서_v0.1]]
  - 보안 요구사항 추적 매트릭스(RTM)

## 5. 수행 절차

### 5.1 사전 준비

1. PM 이 제품 유스케이스·사용자 페르소나·데이터 흐름을 정리한다.
2. R&D 리드가 아키텍처 초안·기술 스택을 준비한다.
3. PSO 가 적용 가능한 표준·법규 체크리스트를 제공한다.

### 5.2 수행 단계

1. **보안 목표 설정**
   - 기밀성·무결성·가용성 목표를 제품 맥락에 맞게 수립(예: 환자 데이터 기밀성 High).
2. **요구사항 도출**
   - SaMD-CSMS 제3~22조 항목을 대조하여 각 항목을 제품 맥락에 번역.
   - 예: 제05조 RBAC → "관리자·임상의·환자 역할 3단계 RBAC 적용".
3. **임상 영향 반영**
   - 각 보안 요구사항이 환자 안전에 미치는 영향을 Safety 리드와 협의하여 우선순위 조정.
4. **요구사항 명세**
   - 각 요구는 INVEST(독립·협상·가치·추정·적정·테스트 가능) 원칙으로 기술.
   - 검증 방법(SAST·DAST·펜테스트·감사) 을 요구별로 명시.
5. **승인**
   - PSO 기술 검토 → VP of R&D 승인.
6. **추적 매트릭스 등록**
   - 요구 ID 를 [[PRO-MDCS-201_사이버보안_리스크_관리_절차_v1.0]] 위협 모델링 및 릴리스 게이트에 연결.

### 5.3 완료 조건

- [ ] 제품 맥락의 보안 목표(CIA) 가 설정되었다
- [ ] SaMD-CSMS 관련 조항별 요구가 맥락화되어 명세되었다
- [ ] 각 요구의 환자 안전 영향이 평가되었다
- [ ] 요구별 검증 방법이 명시되었다
- [ ] PSO 및 VP of R&D 승인이 기록되었다
- [ ] 추적 매트릭스(RTM) 에 요구 ID 가 연결되었다

## 6. 인터페이스 부서

- **임상 담당자**: 환자 안전 영향 평가
- **PSO**: 기술 검토
- **QA**: 검증 방법 피드백
- **법무·RA**: 규제·법규 요구사항

## 7. 주의사항 / 예외 처리

### 7.1 초기 불확실성

시장 조사 초기 단계에서 유스케이스가 유동적인 경우, 핵심 공통 요구(인증·암호화·로그) 를 먼저 확정하고 특화 요구는 Spike 후 보강한다.

### 7.2 규제 변경

개발 중 규제 개정 발생 시 PSO 가 영향 분석을 수행하고, 필요 시 요구사항을 개정. 릴리스 게이트 이전까지 반영.

### 7.3 유스케이스 누락

개발·검증 단계에서 사용성 피드백으로 누락된 유스케이스가 드러나면 요구 추가·변경 관리 절차를 거쳐 반영. 위협 모델링 재실행.

### 7.4 MSP 요구와의 충돌

MSP(병원)별 고유 요구가 조직 표준과 충돌하는 경우 예외 승인 경로로 처리. 예외 요구는 별도 RTM 영역에 기록.
