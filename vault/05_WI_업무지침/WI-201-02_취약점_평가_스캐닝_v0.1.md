# 취약점 평가·스캐닝 업무지침 (WI-201-02)

> 상위 절차: [[PRO-MDCS-201_사이버보안_리스크_관리_절차_v1.0]] · §5 단계 4

## 1. 업무 목적

SaMD-CSMS 제13조 제2호에 따라 디지털의료기기 소프트웨어에 대해 **SAST(정적 분석)·DAST(동적 분석)·SCA(소프트웨어 구성 분석)**를 정기적으로 수행하고, **SBOM 기반 CVE 추적** 체계를 운영하며, 모의 침투 테스트(WI-301-05 연계)와 통합하여 취약점 전수 식별·평가·등록함으로써 제품 사이버보안 수준을 지속 개선한다.

## 2. 수행 주체

- **주 수행자**: PSO, QA (스캐닝 도구 운영·보고)
- **지원**: R&D (코드·빌드 제공, 취약점 조치)
- **검토자**: PSC
- **승인자**: CISO

## 3. 범위

- 자사 개발 코드 및 **제3자·오픈소스** 구성요소 전체 (SBOM 등재 항목)
- **릴리스 전 필수 수행**: SAST, DAST, SCA
- **정기 수행 (분기 1회 이상)**: SCA/SBOM CVE 재스캔, DAST 기능 회귀 스캔
- **연 1회 이상**: 외부 전문기관 모의 침투 테스트 (WI-301-05 연계)
- 시판 후 SBOM 기반 신규 CVE 출현 모니터링 (상시)

## 4. 입력 자료 / 산출물

- **Input**
  - 최신 소스 코드 및 빌드 아티팩트
  - 현행 SBOM (PRO-MDCS-302 에서 관리)
  - NVD·CERT·공급사 보안 권고 피드
  - STRIDE 위협 모델링 결과 (WI-201-01 산출물)
  - 스캐닝 도구 구성 파일 (룰셋·플러그인 버전)
- **Output**
  - SAST 리포트 (취약 코드 위치·등급·CWE 매핑)
  - DAST 리포트 (실행 환경 취약점·OWASP 분류)
  - SCA/SBOM 취약점 보고서 (CVE ID·CVSS·영향 컴포넌트)
  - 통합 취약점 목록 (TMP-RISK-006_취약점_평가_워크시트)
  - CVE 추적 대장 업데이트 (TMP-RISK-007)

## 5. 수행 절차

### 5.1 사전 준비

1. PSO 는 스캐닝 도구 및 룰셋을 최신 버전으로 업데이트하고 도구 유효성 상태(WI-301-06 결과)를 확인한다.
2. R&D 는 분석 대상 코드 브랜치(릴리스 후보 또는 현 프로덕션 버전)와 빌드 아티팩트를 준비하여 PSO 에게 접근 권한을 부여한다.
3. PSO 는 현행 SBOM 에서 타사·오픈소스 컴포넌트 목록을 추출하고 NVD/CVE 데이터베이스와 비교 준비한다.
4. 스캐닝 실행 일정, 참여자, 결과 제출 마감일을 QA·R&D 와 공유한다.

### 5.2 수행 단계

#### A. SAST (정적 분석)

1. **도구 실행**: 승인된 SAST 도구(예: SonarQube, Semgrep, Checkmarx)를 코드베이스에 적용. CWE Top 25 및 의료기기 특화 룰셋(IEC 81001-5-1 대응)을 활성화한다.
2. **결과 분류**: 발견 취약점을 `Critical / High / Medium / Low / Informational` 로 분류. CVSS v3.1 기준 점수를 부여하고 CWE ID 를 매핑한다.
3. **위양성 검토**: QA 와 R&D 가 협력하여 위양성(false positive)을 검토·표시. 위양성 판정 사유를 결과 보고서에 기재한다.
4. **Critical 차단**: Critical 등급 미조치 항목이 있을 경우 릴리스를 차단하고 R&D 에 즉시 통보한다.

#### B. DAST (동적 분석)

1. **테스트 환경 준비**: 프로덕션과 동일 구성의 스테이징 환경에 빌드를 배포. 동적 분석이 프로덕션에 영향을 주지 않도록 격리한다.
2. **도구 실행**: 승인된 DAST 도구(예: OWASP ZAP, Burp Suite Professional)를 사용하여 활성 스캔·스파이더링·인증 우회 테스트를 수행한다.
3. **OWASP Top 10 검증**: OWASP Medical Device Top 10 및 OWASP Web Application Top 10 항목을 체크리스트로 확인한다.
4. **API 엔드포인트 스캔**: 공개 및 내부 API 엔드포인트 전체를 스캔 범위에 포함하고 인증·권한 부여·입력 유효성 검사 취약점을 중점 확인한다.
5. **결과 통합**: SAST 결과와 DAST 결과를 TMP-RISK-006 에 통합하여 중복 제거 및 우선순위를 재산정한다.

#### C. SCA (소프트웨어 구성 분석) 및 SBOM CVE 추적

1. **SCA 도구 실행**: 승인된 SCA 도구(예: Dependency-Track, OWASP Dependency-Check, Snyk)를 현재 SBOM 에 적용한다.
2. **CVE 매핑**: NVD·GitHub Advisory·공급사 권고를 통해 각 컴포넌트의 알려진 CVE 를 조회하고 CVSS 점수를 기록한다.
3. **영향 판단**: 각 CVE 가 제품 기능에서 실제로 악용 가능한지(exploitability in context)를 R&D 와 협력하여 판단한다.
4. **CVE 추적 대장 갱신**: TMP-RISK-007 에 CVE ID·발견일·영향 컴포넌트·CVSS·조치 상태·목표 완료일을 기재한다.
5. **시판 후 상시 모니터링**: PSO 는 NVD RSS·CERT/CC·공급사 보안 뉴스레터를 구독하여 신규 CVE 출현 시 24시간 이내 영향 판단을 수행한다.

#### D. 결과 통합 및 위험 평가 연계

1. SAST·DAST·SCA 결과를 TMP-RISK-006 에 통합하고 중복 취약점을 병합한다.
2. Critical/High 취약점은 WI-201-03(위험 평가 보고서 작성)으로 즉시 연계하여 CVSS × 임상 영향 기반 위험 평가를 수행한다.
3. 조치 불가 취약점(기술적 한계·공급사 미패치 등)은 WI-201-04(잔여 위험 수용)으로 연계한다.
4. 취약점 조치 완료 후 재스캔(retest)을 수행하여 조치 효과를 검증하고 결과를 기록한다.

### 5.3 완료 조건

- [ ] SAST·DAST·SCA 스캔이 모두 수행되고 각 리포트가 생성되었다
- [ ] 발견된 모든 취약점이 TMP-RISK-006 에 등록·분류되었다
- [ ] Critical 등급 취약점의 조치 여부 또는 릴리스 차단 결정이 기록되었다
- [ ] 위양성 항목이 검토·표시되고 사유가 기재되었다
- [ ] SBOM CVE 추적 대장(TMP-RISK-007)이 최신 상태로 갱신되었다
- [ ] Critical/High 취약점이 위험 평가(WI-201-03) 또는 잔여 위험(WI-201-04)으로 연계되었다
- [ ] 조치 완료 항목의 재스캔 결과가 기록되었다
- [ ] PSC 에 취약점 현황 요약이 보고되었다

## 6. 인터페이스 부서

- **R&D**: 코드·빌드·SBOM 제공, 취약점 조치 구현
- **QA**: 스캐닝 도구 운영·결과 검증, 재스캔 수행
- **PSC**: 취약점 평가 결과 검토·승인
- **Safety**: Critical 취약점의 임상 영향 평가
- **IT/운영**: 스테이징 환경 제공, NVD 피드 수신 인프라 지원

## 7. 주의사항 / 예외 처리

### 7.1 스캐닝 도구 유효성 미확인 상태에서 스캔 수행 필요 시

도구 유효성 확인(WI-301-06) 주기가 도래하지 않았거나 도구 교체 직후인 경우, PSO 는 이전 유효성 확인 기록과 현 버전의 릴리스 노트를 검토하여 사용 적합성을 판단하고 사용 근거를 취약점 보고서 커버 페이지에 기재한다.

### 7.2 공급사 미패치 CVE(0-day 또는 장기 미조치)

공급사가 패치를 제공하지 않은 CVE 의 경우, PSO 는 해당 취약점의 제품 맥락 내 악용 가능성을 분석하고 보완 통제(완화 조치: 네트워크 격리·입력 검증 강화·기능 비활성화 등)를 설계한다. 잔여 위험은 WI-201-04 경로로 PSC/CEO 수용 결정을 받는다. 공급사 패치 공개 여부를 TMP-RISK-007 에서 분기별로 추적한다.

### 7.3 스캐닝 중 프로덕션 서비스 장애 유발

DAST 스캔이 예상치 못하게 프로덕션 또는 공유 스테이징 서비스에 영향을 미치는 경우, 즉시 스캔을 중단하고 서비스 복구를 우선한다. 영향 범위를 기록하고 원인 분석 후 격리 환경에서 재수행한다. 재발 방지를 위해 스캔 대상 환경 격리 절차를 개선한다.

### 7.4 릴리스 일정 압박으로 스캔 생략 요청

릴리스 일정상의 이유로 스캔 생략을 요청받는 경우, PSO 는 해당 요청을 거부하고 PSC 에 보고한다. 불가피하게 단축이 필요하면 PSC 가 스캔 범위 축소와 보완 일정을 승인하고, 완전 스캔 완료 전까지의 위험 수용을 기록한다.
