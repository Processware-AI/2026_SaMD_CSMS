# 실행파일 차단 업무지침 (WI-403-02)

> 상위 절차: [[PRO-MDCS-403_데이터_및_파일_보안_관리_절차_v1.0]] · §5 단계 3

## 1. 업무 목적

사용자·외부 시스템이 업로드하는 파일 중 실행 가능한 바이너리·스크립트를 서버에 저장·실행되기 전에 원천 차단하여 웹쉘 설치, 원격 코드 실행(RCE), 랜섬웨어 배포 등의 보안 사고를 예방한다. 업무상 실행 파일 처리가 불가피한 경우 샌드박스 검증 및 CISO 예외 승인을 거쳐 화이트리스트를 관리한다. SaMD-CSMS 제07조 제3호의 실행 파일 차단 요건(MDCS-R-073)을 충족한다.

## 2. 수행 주체

- **주 수행자**: 개발팀 보안담당 (차단 로직 설계·구현, 화이트리스트 코드 반영)
- **검토자**: SecOps (차단 정책 실효성 검증, 우회 시도 탐지)
- **승인자**: CISO (예외 허용 최종 승인)
- **QA**: 분기 침투 테스트를 통한 차단 로직 우회 검증

## 3. 범위

- 모든 파일 업로드 엔드포인트(REST API, 웹 폼, FTP, SFTP, 파일 공유 서비스)
- 차단 대상 확장자: `.exe`, `.dll`, `.bat`, `.cmd`, `.ps1`, `.vbs`, `.js`, `.jar`, `.war`, `.ear`, `.sh`, `.py`, `.rb`, `.pl`, `.php`, `.asp`, `.aspx`, `.cgi`, `.com`, `.scr`, `.pif`, `.msi`, `.cab`, `.bin` (및 조직이 별도 지정하는 추가 확장자)
- 서버 측(백엔드) 파일 저장 경로 및 웹 루트 접근 가능 디렉터리
- 예외 범위: 개발·빌드 파이프라인 전용 내부 아티팩트 저장소 (별도 접근통제 적용)

## 4. 입력 자료 / 산출물

- **Input**
  - 파일 업로드 요구사항 정의서 (허용 파일 유형·최대 크기·업로드 목적)
  - [[TMP-DEV-021_실행파일_차단_예외_신청서_v0.1]] (예외 허용 신청 시)
  - OWASP Unrestricted File Upload 가이드라인 최신본
- **Output**
  - 차단 로직 소스코드·설정 파일 (확장자 필터, 매직넘버 검증, Content-Type 검증)
  - [[TMP-DEV-022_실행파일_화이트리스트_대장_v0.1]] (예외 허용 파일 유형·적용 범위·승인자·만료일)
  - SAST/DAST 차단 로직 검증 리포트
  - 분기 침투 테스트 결과 (차단 우회 시도 결과)

## 5. 수행 절차

### 5.1 사전 준비

1. 개발팀 보안담당은 서비스별 파일 업로드 엔드포인트 목록을 식별하고, 각 엔드포인트가 허용해야 하는 정상 파일 유형(예: `.pdf`, `.jpg`, `.png`, `.csv`)을 서비스 소유자와 확인한다.
2. 차단 대상 확장자 목록을 [[TMP-DEV-022_실행파일_화이트리스트_대장_v0.1]] 에 기재하고, CISO 초기 승인을 받는다.
3. 서버 측 파일 저장 디렉터리가 웹 루트 외부에 위치하거나 실행 권한(executable bit)이 제거되어 있음을 확인한다.

### 5.2 수행 단계

1. **확장자 검증 (1차 차단)**
   - 파일 업로드 요청 수신 시, 파일명에서 추출한 확장자(복수 확장자 포함, 예: `malware.php.jpg`)를 허용 확장자 화이트리스트와 대조한다.
   - 화이트리스트에 없는 확장자이거나 차단 목록에 해당하는 경우 즉시 거부(HTTP 400/415)하고 업로드 시도 이벤트를 로그에 기록한다.
   - 확장자 우회 시도를 방지하기 위해 파일명 정규화(URL 인코딩 해제, null byte 제거, 대소문자 통일)를 수행한 후 검증한다.
2. **매직 넘버 (Magic Number) 검증 (2차 차단)**
   - 파일 헤더(최초 수백 바이트)를 읽어 실제 파일 형식의 매직 넘버와 비교한다. 확장자와 실제 파일 형식이 불일치하는 경우 차단한다.
   - 허용 파일 유형별 매직 넘버 목록을 코드 상수 또는 설정 파일로 관리하며, 라이브러리 업데이트 시 함께 갱신한다.
3. **Content-Type 및 MIME 검증 (3차 차단)**
   - HTTP 요청의 `Content-Type` 헤더가 허용 MIME 타입 목록에 포함되어 있는지 검증한다.
   - 클라이언트 제공 `Content-Type` 만을 신뢰하지 않고, 서버 측 MIME 탐지 라이브러리로 재검증한다.
4. **저장 경로 및 파일명 무해화 (Sanitization)**
   - 업로드된 파일을 원본 파일명 그대로 저장하지 않고, UUID 기반으로 파일명을 재생성하여 저장한다.
   - 저장 경로는 웹 서버 문서 루트 밖으로 설정하며, 다운로드는 전용 API를 통해서만 제공한다.
   - 저장된 파일이 위치하는 디렉터리에 실행 권한이 없음을 배포 파이프라인 단계에서 자동 검증한다.
5. **악성 코드 스캔 (4차 검증)**
   - 파일 저장 직후 또는 저장 전, 안티바이러스·안티멀웨어 스캐닝 엔진(ClamAV 또는 통합 보안 플랫폼)을 통해 스캔을 수행한다.
   - 위협 탐지 시 파일을 격리(quarantine) 처리하고 SecOps에 즉시 알림을 발송한다.
6. **화이트리스트 예외 처리**
   - 업무상 실행 파일 처리가 불가피한 경우(예: 의료기기 펌웨어 배포 전용 내부 포털), 개발팀 보안담당이 [[TMP-DEV-021_실행파일_차단_예외_신청서_v0.1]] 를 작성하고 CISO 승인을 받는다.
   - 승인된 예외는 [[TMP-DEV-022_실행파일_화이트리스트_대장_v0.1]] 에 적용 범위(특정 엔드포인트·역할·IP)·만료일·추가 통제(샌드박스 의무화, 감사 로그 강화)를 명시하여 등록한다.
   - 예외 유효기간은 최대 1년이며, 연장 시 재신청·재승인이 필요하다.
7. **분기 검증 (침투 테스트 / SAST)**
   - QA 또는 SecOps가 분기 1회 차단 로직 우회 시도(예: 이중 확장자, null byte, zip slip, polyglot 파일)를 수행하고 결과를 보고한다.
   - SAST 도구가 업로드 처리 코드의 취약점을 CI/CD 파이프라인에서 매 빌드 시 검사한다.

### 5.3 완료 조건

- [ ] 모든 파일 업로드 엔드포인트에 확장자·매직넘버·MIME 3단계 검증이 구현되었다
- [ ] 차단 로직이 SAST에서 Critical/High 결함 없이 통과되었다
- [ ] 저장 경로가 웹 루트 외부이고 실행 권한이 제거되었음이 확인되었다
- [ ] 업로드 차단 이벤트가 로그에 기록되고 SecOps 모니터링 대상에 포함되었다
- [ ] 화이트리스트 예외는 CISO 승인과 만료일이 대장에 기재되었다
- [ ] 분기 침투 테스트에서 우회 성공 건수 0건이 확인되었다

## 6. 인터페이스 부서

- **QA**: 릴리스 전 차단 로직 기능 테스트, 분기 침투 테스트 수행
- **SecOps**: 업로드 차단 이벤트·악성 파일 탐지 이벤트 실시간 모니터링 및 대응
- **CISO**: 화이트리스트 예외 최종 승인
- **서비스 소유자 / 제품팀**: 허용 파일 유형 요구사항 정의, 예외 신청
- **DevOps / 인프라팀**: 저장 경로 권한 설정, CI/CD SAST 연동

## 7. 주의사항 / 예외 처리

### 7.1 Polyglot 파일 (다중 형식 파일) 우회 시도

하나의 파일이 두 가지 이상의 형식으로 해석될 수 있는 Polyglot 파일(예: JPEG이면서 동시에 유효한 PHP 스크립트)은 확장자·MIME 단일 검증만으로는 차단이 불완전하다. 이에 대응하기 위해 파일 파싱 라이브러리를 사용하여 파일의 논리적 구조가 선언된 형식과 일치하는지 심층 검증(deep content validation)하며, 저장 디렉터리 실행 권한 제거와 병행하여 이중 방어를 유지한다.

### 7.2 악성 파일 업로드 탐지 후 포렌식 보존

업로드 시도된 악성 파일이 탐지된 경우, 즉시 격리하되 파일을 완전 삭제하지 않고 격리 스토리지에 해시값과 함께 최소 90일간 보존한다. SecOps가 격리된 파일 정보를 포렌식 조사에 활용하고, 침해사고 대응 프로세스([[PRO-MDCS-501]] 등 연계)와 연계하여 공격자 경로를 분석한다.

### 7.3 레거시 시스템 통합 시 실행 파일 교환이 불가피한 경우

레거시 의료기기와의 통합에서 펌웨어 업데이트 파일(.bin, .hex 등)을 업로드해야 하는 경우, 해당 엔드포인트를 내부 네트워크 전용으로 격리하고, 파일 서명 검증(코드 서명 인증서 확인)을 의무화하며, 모든 업로드 이벤트를 강화 감사 로그(enhanced audit log)로 별도 관리한다. 이 예외는 반드시 화이트리스트 대장에 등록하고 연 1회 재검토한다.
