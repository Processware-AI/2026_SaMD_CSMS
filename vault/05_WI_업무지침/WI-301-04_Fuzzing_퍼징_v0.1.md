# Fuzzing(퍼징) 업무지침 (WI-301-04)

> 상위 절차: [[PRO-MDCS-301_보안_SDLC_절차_v1.0]] · §5 단계 8

## 1. 업무 목적

SaMD-CSMS 제14조 제2호에 따라 자동화된 퍼징(Fuzzing) 도구를 활용하여 디지털의료기기 소프트웨어의 **파서·입력 처리 함수·네트워크 인터페이스·파일 처리 모듈**에 비정상 입력·경계값·무작위 변형 데이터를 주입하고, 충돌(Crash)·메모리 오류·예기치 않은 동작을 탐지하여 환자 안전에 영향을 줄 수 있는 결함을 출시 전에 제거한다.

## 2. 수행 주체

- **주 수행자**: QA (퍼징 캠페인 실행·결과 분류·보고)
- **지원**: R&D (퍼징 드라이버 작성·결함 분석·조치)
- **검토자**: PSO (결과 검토, 보안 영향 판단)
- **승인자**: VP of R&D

## 3. 범위

- **필수 대상** (릴리스 전 수행):
  - 외부 입력을 처리하는 모든 파서 (의료 데이터 포맷: HL7, DICOM, FHIR JSON)
  - 네트워크 통신 처리 모듈 (TCP/UDP 수신부, REST API 역직렬화 코드)
  - 파일 업로드·처리 함수 (이미지·PDF·설정 파일 파서)
  - CLI/UI 입력 처리 경로
- **권고 대상** (회귀 퍼징, 스프린트 주기):
  - 이전 릴리스에서 결함이 발견된 컴포넌트
  - 코드 변경이 발생한 입력 처리 모듈
- AI 적용 기기의 경우 추론 입력(inference input) 처리 경로 추가 포함

## 4. 입력 자료 / 산출물

- **Input**
  - 빌드 아티팩트 (퍼징 대상 바이너리 또는 라이브러리)
  - 입력 코퍼스 (corpus): 정상 샘플 입력 파일·패킷
  - 퍼징 드라이버(harness) 코드 (R&D 작성)
  - 이전 캠페인 크래시 데이터 (회귀 퍼징 시)
  - 퍼징 도구 설정 파일 (사전·뮤테이터·타임아웃 설정)
- **Output**
  - 퍼징 리포트 (TMP-TEST-004): 캠페인 요약·크래시 목록·재현 입력·중복 제거 결과
  - 크래시 재현 입력 파일 (아카이브)
  - 결함 티켓 (이슈 트래커 등록)
  - 코드 커버리지 보고서 (측정 가능한 경우)

## 5. 수행 절차

### 5.1 사전 준비

1. **퍼징 도구 유효성 확인**: WI-301-06 에 따른 최근 반기 유효성 확인 기록을 검토하고 승인된 도구 버전임을 확인한다.
2. **빌드 설정**: AddressSanitizer(ASan)·UndefinedBehaviorSanitizer(UBSan) 등 **메모리 안전 계측기(sanitizer)**를 활성화하여 퍼징용 빌드를 생성한다. 계측기 없는 빌드로 퍼징 수행 시 탐지율이 현저히 낮아진다.
3. **퍼징 드라이버 작성**: R&D 는 각 퍼징 대상 함수/모듈에 대해 `LLVMFuzzerTestOneInput` (libFuzzer) 또는 AFL 래퍼 형식의 드라이버를 작성한다. 드라이버는 대상 함수만 호출하고 부작용(파일 I/O·네트워크·DB 접근)을 최소화한다.
4. **코퍼스 준비**: 퍼징 대상 형식의 유효 샘플 입력 파일(예: 실제 DICOM 이미지 익명화본, HL7 메시지 샘플)을 최소 50개 이상 수집하여 코퍼스 디렉토리에 배치한다.
5. **실행 환경 격리**: 퍼징은 프로덕션·스테이징 환경과 완전히 격리된 퍼징 전용 VM 또는 컨테이너에서 수행한다.

### 5.2 수행 단계

#### A. 퍼징 캠페인 실행

1. **도구 선택**: 대상 특성에 따라 도구를 선택한다.
   - **AFL++ (American Fuzzy Lop++)**: 바이너리·C/C++ 코드 커버리지 기반 뮤테이션 퍼징
   - **libFuzzer**: LLVM 기반 인-프로세스 퍼징, clang 빌드 환경에 최적
   - **Jazzer**: JVM 기반 코드(Java/Kotlin)의 퍼징
   - **Radamsa**: 구조화된 입력의 블랙박스 뮤테이션 (드라이버 없이 사용 가능)
2. **캠페인 파라미터 설정**: 타임아웃(단일 입력 최대 처리 시간), 메모리 제한, 최대 실행 횟수(또는 시간)를 설정한다. 권고 최소 캠페인 시간: 24시간 지속 실행 (릴리스 전).
3. **병렬 실행**: 가능한 경우 CPU 코어 수에 따라 병렬 퍼저 인스턴스를 실행하여 탐색 속도를 높인다.
4. **실행 중 모니터링**: 퍼저 실행 중 크래시 발생 즉시 크래시 입력이 자동 저장되는지 확인한다. 장시간 신규 커버리지 증가가 없으면 코퍼스나 뮤테이터 전략을 조정한다.

#### B. 크래시 분류 및 중복 제거

1. **크래시 수집**: 캠페인 종료 후 크래시 디렉토리에서 모든 크래시 입력을 수집한다.
2. **중복 제거**: 동일한 코드 위치에서 발생한 크래시를 스택 트레이스 기반으로 중복 제거하여 고유 결함 수를 산정한다.
3. **재현 확인**: 각 고유 크래시에 대해 정상 빌드(sanitizer 포함)에서 재현 가능한지 확인한다.
4. **보안 영향 분류**: 각 크래시를 다음 등급으로 분류한다.
   - **Critical**: 원격 코드 실행(RCE) 가능 메모리 취약점(힙 버퍼 오버플로, UAF)
   - **High**: 프로세스 크래시(서비스 거부·DoS), 스택 오버플로
   - **Medium**: 무한 루프·자원 고갈(메모리 누수·CPU 스핀)
   - **Low**: 어설션 실패·예외 미처리 (실제 악용 가능성 낮음)

#### C. 결함 보고 및 추적

1. **이슈 등록**: Critical·High 결함을 이슈 트래커에 등록하고 크래시 재현 입력 파일을 첨부한다. 이슈에는 결함 유형·발생 위치(파일·줄번호)·스택 트레이스·보안 영향 등급·발견 도구를 기재한다.
2. **TMP-TEST-004 작성**: 캠페인 전체 결과를 퍼징 리포트 양식에 기재한다. 실행 통계(총 실행 횟수·소요 시간·코드 커버리지·고유 크래시 수)와 분류 결과를 포함한다.
3. **조치 의뢰**: Critical·High 결함은 R&D 에 즉시 통보하고 조치 기한(Critical: 릴리스 차단·7일 이내, High: 릴리스 전 조치)을 부여한다.
4. **재퍼징(Retest)**: 결함 조치 후 동일 입력으로 재현 확인 및 동일 대상에 대한 추가 퍼징을 수행하여 조치 효과를 검증한다.

#### D. 위험 연계

1. Critical·High 결함이 조치되지 않고 릴리스 게이트로 진행되는 경우, PRO-MDCS-201 §5 단계 7~8 절차(잔여 위험 수용, WI-201-04)로 연계한다.
2. 퍼징 결과를 취약점 목록(TMP-RISK-006)에 통합한다.

### 5.3 완료 조건

- [ ] 필수 퍼징 대상(파서·네트워크·파일 처리 모듈)에 대한 캠페인이 수행되었다
- [ ] 각 캠페인이 sanitizer 활성화 빌드에서 최소 24시간 실행되었다
- [ ] 발견된 크래시가 중복 제거되고 보안 영향 등급이 분류되었다
- [ ] Critical·High 결함이 이슈 트래커에 등록되었다
- [ ] TMP-TEST-004 퍼징 리포트가 작성·서명되었다
- [ ] Critical 결함의 조치 또는 릴리스 차단 결정이 기록되었다
- [ ] 조치 완료 결함의 재퍼징 결과가 기록되었다
- [ ] 퍼징 결과가 취약점 목록(TMP-RISK-006)에 통합되었다

## 6. 인터페이스 부서

- **R&D**: 퍼징 드라이버 작성, 크래시 원인 분석, 결함 조치
- **PSO**: 퍼징 결과의 보안 영향 판단, 잔여 위험 연계
- **QA**: 퍼징 도구 운영, 결과 보고서 작성, 릴리스 게이트 제출
- **IT/인프라**: 퍼징 전용 격리 환경 제공, CPU 리소스 할당

## 7. 주의사항 / 예외 처리

### 7.1 Sanitizer 활성화 빌드와 프로덕션 빌드 결과 불일치

Sanitizer 빌드에서 발견된 크래시가 프로덕션 빌드에서 재현되지 않는 경우가 있다. 이 경우 R&D 는 컴파일러 최적화 수준 및 메모리 레이아웃 차이를 분석하여 취약점 실존 여부를 판단한다. 재현 불가 확인이 되기 전까지는 취약점으로 간주하여 추적한다.

### 7.2 드라이버 작성 시 대상 모듈의 외부 의존성(DB·네트워크) 분리 불가

대상 모듈이 DB·파일시스템·네트워크에 강결합되어 드라이버 분리가 어려운 경우, Mock·Stub 을 사용하거나 인-메모리 파일시스템을 적용한다. 분리가 전혀 불가한 경우 전체 서비스 수준의 블랙박스 퍼징(Radamsa + DAST 연계)으로 대체하고 대체 사유를 리포트에 기재한다.

### 7.3 캠페인 실행 중 퍼징 환경 자원 고갈 (OOM·디스크 풀)

메모리 부족 또는 크래시 파일 누적으로 퍼징 환경이 중단된 경우, 진행 상태를 저장하고 자원 추가 후 재개한다. 이미 수집된 크래시 데이터는 보존하여 분석에 활용한다. 빈번히 발생하면 메모리 제한 파라미터와 크래시 보관 정책을 조정한다.

### 7.4 레거시 언어(C/C++ 외) 또는 해석형 언어(Python·JavaScript) 대상 퍼징

Python·Node.js 등 해석형 언어로 작성된 모듈의 경우, Atheris(Python)·jest-fuzz·Jazzer(JVM) 등 언어별 퍼징 프레임워크를 사용하거나, 인터프리터 수준에서 AFL 커버리지 계측을 적용한다. 도구 적합성은 WI-301-06 유효성 확인 시 사전 검증한다.
