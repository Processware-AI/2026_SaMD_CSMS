# 세션 및 쿠키 관리 업무지침 (WI-401-02)

> 상위 절차: [[PRO-MDCS-401_접근통제_및_인증_관리_절차_v1.0]] · §5 단계 5

## 1. 업무 목적

인증 이후 사용자 세션이 탈취·고정·도용되지 않도록 세션 생명주기를 안전하게 관리하고, 쿠키에 보안 속성을 적용하여 XSS·CSRF·세션 고정 공격 등의 위협을 방지한다. 본 지침은 SaMD-CSMS 제05조 제2호에서 요구하는 세션·쿠키 보안 요건을 디지털의료기기 웹·앱 인터페이스에 적용하는 구체적 기준을 제시한다.

## 2. 수행 주체

- **주 수행자**: 시스템 소유자 (세션·쿠키 정책을 시스템에 구현·유지)
- **검토자**: IAM 관리자 (세션 정책 기준 제공·로그 분석), SecOps (이상 세션 탐지)
- **승인자**: 오동석 (정책 기준 변경·예외 승인)
- **협력**: 개발팀 (프레임워크 레벨 구현), 인프라 팀 (로드밸런서·리버스프록시 세션 설정)

## 3. 범위

- 당사 디지털의료기기 제품의 웹 관리 콘솔, REST API 클라이언트, 모바일 앱 인터페이스
- 운영 환경(Production), 스테이징(Staging), 테스트 환경의 외부 접근 가능 엔드포인트
- 세션 쿠키(Session Cookie), JWT(JSON Web Token), OAuth 2.0 액세스 토큰·리프레시 토큰

## 4. 입력 자료 / 산출물

- **Input**
  - 시스템별 인증 아키텍처 명세 (세션 관리 방식: 쿠키 기반 세션 vs. 토큰 기반)
  - 사용자 유형별 업무 특성 (세션 타임아웃 기준 수립 근거)
  - 보안 테스트 결과 (침투 테스트·정적 분석에서 도출된 세션 관련 취약점)
- **Output**
  - 세션·쿠키 정책 설정 명세서 (시스템별 파라미터 값 포함)
  - 정책 적용 확인 로그 (구성 감사 스크립트 실행 결과)
  - 세션 이상 탐지 알림 설정 기록
  - 세션 정책 위반 또는 예외 승인 이력

## 5. 수행 절차

### 5.1 사전 준비

1. 시스템 소유자가 관리하는 각 시스템의 세션 관리 방식(쿠키 기반 vs. 토큰 기반)을 파악하고, 적용해야 할 정책 파라미터 목록을 본 지침 §5.2를 기준으로 준비한다.
2. 개발팀과 인프라 팀에 정책 변경 적용 계획을 공유하고, 변경 적용 후 회귀 테스트 일정을 조율한다.
3. SecOps에게 비정상 세션 탐지 기준(동일 세션 ID의 다중 IP 접속, 단시간 과도한 요청 등)을 알림 규칙으로 추가하도록 요청한다.

### 5.2 수행 단계

#### 5.2.1 세션 타임아웃 설정

1. **Idle(비활성) 타임아웃** 기준을 시스템·사용자 역할별로 다음과 같이 적용한다.
   - 관리자·특권 계정: 15분 비활성 후 세션 만료
   - 일반 운영자 계정: 30분 비활성 후 세션 만료
   - 의료진 임상 인터페이스: 임상 환경 특성을 고려하여 시스템 소유자가 PSO와 협의하여 결정 (기본 30분, 임상 위험 고려 시 60분까지 허용)
   - API·서비스 간 세션: 비활성 타임아웃 15분, 절대 만료(Absolute Timeout) 8시간
2. **절대 만료(Absolute Timeout)** 를 설정하여 비활성 여부와 무관하게 세션이 영원히 유지되지 않도록 한다.
   - 웹 관리 콘솔·임상 인터페이스: 절대 만료 8시간
   - 관리자 세션: 절대 만료 4시간
3. 세션 만료 시 사용자에게 만료 3분 전 경고 팝업을 표시하고, 만료 후 재인증 없이 접근 시 로그인 페이지로 리다이렉트한다.

#### 5.2.2 세션 고정(Session Fixation) 공격 방지

1. **인증 성공 직후 반드시 새 세션 ID를 발급**한다. 로그인 전 세션 ID를 로그인 후에도 재사용하는 것을 금지한다.
2. 세션 ID는 암호학적으로 안전한 난수 생성기(CSPRNG)를 사용하여 최소 128비트 엔트로피를 보장한다.
3. 세션 ID를 URL 파라미터로 전달하는 방식을 금지한다. 모든 세션 ID는 쿠키 또는 Authorization 헤더를 통해서만 전달한다.

#### 5.2.3 쿠키 보안 속성 설정

세션 쿠키에 다음 보안 속성을 필수 적용한다.

| 속성 | 설정 값 | 목적 |
|---|---|---|
| `HttpOnly` | 필수 적용 | JavaScript에서 쿠키 접근 금지 (XSS 탈취 방지) |
| `Secure` | 필수 적용 (HTTPS 강제) | HTTPS 연결에서만 쿠키 전송 (평문 전송 방지) |
| `SameSite` | `Strict` 또는 `Lax` | CSRF 공격 방지. 외부 도메인 요청에 쿠키 미전송 |
| `Max-Age` / `Expires` | Idle 타임아웃과 일치 | 브라우저 재시작 후에도 만료 기준 적용 |
| `Path` | 최소 필요 경로 | 쿠키 접근 범위 제한 |
| `Domain` | 명시적 설정 (서브도메인 와일드카드 금지) | 불필요한 도메인 공유 방지 |

- 민감 데이터(사용자 식별자·권한 정보)를 쿠키에 직접 저장하지 않는다. 서버측 세션 저장소의 세션 ID 참조만 쿠키에 저장한다.
- `SameSite=None` 설정은 반드시 `Secure` 속성과 함께만 사용 가능하며, 오동석 예외 승인이 있는 경우에만 허용한다.

#### 5.2.4 JWT 및 OAuth 2.0 토큰 관리

1. **액세스 토큰 만료 기간**: 최대 15분. 만료 후 리프레시 토큰으로 재발급 받아야 한다.
2. **리프레시 토큰 만료 기간**: 최대 24시간 (관리자용 8시간). 리프레시 토큰은 1회 사용 후 폐기하는 **리프레시 토큰 로테이션** 방식을 적용한다.
3. JWT의 알고리즘 필드(`alg`)는 `RS256` 또는 `ES256` 이상의 비대칭 알고리즘을 사용한다. `alg: none` 또는 대칭 알고리즘(`HS256`)을 서버-클라이언트 간 공개 API에 사용하는 것을 금지한다.
4. 토큰 폐기(Revocation) 메커니즘을 구현한다: 로그아웃·비밀번호 변경·계정 비활성화 이벤트 발생 시 해당 토큰을 즉시 폐기 목록(Blocklist 또는 短命 TTL 캐시)에 등록한다.
5. JWT 페이로드에 민감한 PII(개인식별정보) 또는 의료 데이터를 포함하지 않는다.

#### 5.2.5 중복 로그인 통제

1. 동일 계정의 동시 세션 허용 여부를 사용자 역할별로 설정한다.
   - 관리자·특권 계정: 동시 세션 1개 허용 (새 로그인 시 기존 세션 강제 종료)
   - 일반 운영자: 동시 세션 최대 3개 허용, 임계값 초과 시 오래된 세션 자동 종료
2. 새 로그인으로 기존 세션이 강제 종료되는 경우, 기존 세션 사용자에게 세션 종료 사유와 최근 접속 IP를 포함한 알림을 발송한다.

#### 5.2.6 정책 적용 확인 및 감사

1. 분기 1회 구성 감사 스크립트를 실행하여 모든 엔드포인트의 쿠키 속성과 세션 타임아웃 값이 본 지침의 기준을 충족하는지 확인한다.
2. 감사 결과를 IAM 관리자에게 보고하고, 미충족 항목은 30일 이내 시정한다.

### 5.3 완료 조건

- [ ] 모든 관리·운영 인터페이스에서 Idle 타임아웃이 역할별 기준(관리자 15분, 일반 30분)으로 설정되어 있다
- [ ] 모든 세션 쿠키에 `HttpOnly`, `Secure`, `SameSite` 속성이 설정되어 있다
- [ ] 인증 성공 직후 세션 ID 재발급이 구현되어 있고 세션 고정 방지 테스트를 통과하였다
- [ ] JWT 액세스 토큰 만료 기간이 15분 이하로 설정되어 있고 리프레시 토큰 로테이션이 구현되어 있다
- [ ] `alg: none` 또는 HS256 알고리즘 사용이 없음이 확인된다
- [ ] 로그아웃·비밀번호 변경 시 토큰 폐기 메커니즘이 동작한다
- [ ] 관리자 계정의 동시 세션이 1개로 제한되어 있다
- [ ] 분기 구성 감사 결과가 기록되고, 미충족 항목에 대한 시정 계획이 있다

## 6. 인터페이스 부서

- **개발팀**: 세션·쿠키·JWT 정책을 애플리케이션 프레임워크 수준에서 구현
- **인프라 팀**: 로드밸런서·리버스프록시의 세션 관련 헤더 설정, HTTPS 강제 리다이렉트
- **IAM 관리자**: 세션 정책 기준 제공, 세션 이상 로그 분석, 구성 감사 보고서 수신
- **SecOps**: 비정상 세션 활동(다중 IP 사용·세션 하이재킹 징후) 탐지 알림 운영
- **오동석**: 정책 기준 변경 및 `SameSite=None` 등 예외 승인

## 7. 주의사항 / 예외 처리

### 7.1 레거시 브라우저 호환성으로 인한 SameSite 속성 미지원

구형 브라우저에서 `SameSite=Strict` 또는 `SameSite=Lax`를 인식하지 못하는 경우 쿠키가 차단되어 정상 동작하지 않을 수 있다. 이 경우 서버측 CSRF 토큰(Synchronizer Token Pattern)을 추가 구현하여 SameSite 속성을 대체하는 보상 통제를 적용하고, 지원 중단 브라우저 버전을 공식 호환성 정책에서 제외한다. 예외 처리 내용을 PSO에 보고하고 오동석 승인을 받는다.

### 7.2 외부 제3자 SSO(Single Sign-On) 연동 시 세션 관리

외부 IdP(Identity Provider, 예: SAML·OIDC)와 SSO 연동 시 세션 관리 주체가 분산된다. 이 경우 다음 원칙을 적용한다: IdP 세션이 만료되어도 서비스 측 세션이 독립적으로 만료되도록 로컬 세션 타임아웃을 별도 설정한다. IdP 로그아웃 이벤트(Back-Channel Logout) 수신 시 서비스 측 세션을 즉시 종료하는 로직을 구현한다. SSO 연동 구조가 변경될 때마다 PSO에 영향 분석을 수행하도록 요청한다.

### 7.3 임상 응급 상황에서의 세션 타임아웃 이슈

임상 현장에서 의료진이 응급 상황 처리 중 세션이 만료되어 재인증이 요구되는 경우 환자 안전에 직접 영향을 미칠 수 있다. 이 경우 의료 기관 운용 담당자가 PSO에 예외 적용을 요청하여 특정 임상 기능 인터페이스에 한해 타임아웃 기간을 연장(최대 60분)할 수 있다. 예외 적용 시 반드시 해당 인터페이스에 대한 추가 접근 로깅을 강화하고 SecOps에 알림을 추가하며, 오동석가 예외 유효기간(최대 6개월)을 승인한다.

### 7.4 세션 하이재킹 탐지 시 대응

SecOps가 동일 세션 ID의 비정상적인 다중 IP 접속(예: 5분 이내 지리적으로 불가능한 IP 전환)을 탐지하는 경우, 해당 세션을 즉시 강제 종료하고 IAM 관리자에게 알림을 발송한다. IAM 관리자는 계정 소유자에게 비밀번호 변경 및 모든 활성 세션 종료를 요청하고, 침해 징후가 있는 경우 [[PRO-MDCS-502_침해사고_대응_절차]] 를 개시한다.

- 세션·쿠키 정책 설정 명세서: 시스템별 구성 문서 저장소
- 구성 감사 결과: IAM 관리자 보고서 (분기별)
