# HSM/KMS 운영 업무지침 (WI-402-05)

> 상위 절차: [[PRO-MDCS-402_보안_통신_및_암호화_관리_절차_v1.0]] · §5 단계 7~8

## 1. 업무 목적

HSM 및 KMS를 안전하게 운영하여 암호화 키의 생성·보관·교체·폐기 전 수명주기를 통제하고, 키 접근을 최소권한·MFA 기반으로 제한하며, 모든 접근 이력을 감사 가능하게 유지함으로써 키 노출·오용·분실로 인한 데이터 유출 및 서비스 중단을 예방한다. SaMD-CSMS 제09조 제3호의 HSM/TPM/KMS 및 보안 부팅 요건, 제09조 제1·2호의 키 분리·로테이션·접근·백업 요건을 충족한다.

## 2. 수행 주체

- **주 수행자**: KMS Admin (키 생성·로테이션·폐기, HSM 운영)
- **검토자**: 오동석 (키 정책 승인, 예외 승인)
- **보조 수행자**: 인프라팀 (HSM 하드웨어 유지보수, 네트워크 연결)
- **감사자**: SecOps (접근 로그 이상 탐지), 내부감사팀 (연 1회 감사)

## 3. 범위

- 조직이 운영하는 모든 HSM 장비(온프레미스 FIPS 140-3 인증 장비, 클라우드 HSM 포함)
- KMS 소프트웨어/서비스(자체 운영 또는 CSP 관리형 KMS)
- 대칭키(AES-256), 비대칭키(RSA-4096, ECC P-384 이상), MAC 키, TLS 인증서 키
- 키 수명주기 전 단계: 생성 → 배포 → 운용 → 교체(로테이션) → 보관(보존) → 폐기
- HSM 보안 부팅, 펌웨어 무결성 검증, 물리 보안

## 4. 입력 자료 / 산출물

- **Input**
  - 암호화 정책 및 알고리즘 목록 (PRO-MDCS-402 §5 단계 5 산출물)
  - 시스템별 키 사용 요구서 (개발팀 또는 서비스 소유자 제출)
  - HSM 제조사 보안 가이드 및 FIPS 140-3 운영 매뉴얼
- **Output**
  - [[TMP-KMS-001_암호키_인벤토리_대장_v0.1]] (키 ID·알고리즘·용도·유효기간·소유자·상태)
  - [[TMP-KMS-002_HSM_접근_감사_로그_v0.1]] (접근 주체·시각·작업·결과)
  - [[TMP-KMS-003_키_로테이션_체크리스트_v0.1]] (로테이션 사전/사후 검증 항목)
  - HSM 보안 부팅·펌웨어 무결성 검증 기록
  - 폐기 완료 증적 (키 폐기 확인서)

## 5. 수행 절차

### 5.1 사전 준비

1. KMS Admin은 [[TMP-KMS-001_암호키_인벤토리_대장_v0.1]] 을 최신 상태로 유지하며, 키별 유효기간·로테이션 주기·소유 서비스를 명시한다.
2. HSM 장비의 FIPS 140-3 인증 상태와 펌웨어 버전을 분기 1회 확인하고, 취약점 패치 출시 시 30일 이내 적용 계획을 수립한다.
3. KMS 접근 계정은 Dual Control(두 명의 KMS Admin 공동 승인) 원칙을 적용하며, MFA 인증이 설정된 전용 계정만 사용한다.
4. 비상 복구용 키 백업본은 물리적으로 분리된 보안 금고(오프사이트 포함)에 보관하며, 백업본 위치·접근 권한자 목록을 대장에 기록한다.

### 5.2 수행 단계

1. **키 생성**
   - 서비스 소유자가 키 사용 요구서(알고리즘·용도·유효기간·대상 시스템)를 KMS Admin에게 제출한다.
   - KMS Admin이 HSM의 내부 난수 생성기(TRNG)를 사용하여 키를 생성한다. 외부 소프트웨어 난수 생성기 사용은 금지한다.
   - 생성 직후 키를 [[TMP-KMS-001_암호키_인벤토리_대장_v0.1]] 에 등록하며, 키 ID·알고리즘·용도·생성 일시·유효기간·소유자를 기재한다.
   - 생성 이벤트가 [[TMP-KMS-002_HSM_접근_감사_로그_v0.1]] 에 자동 기록됨을 확인한다.
2. **키 배포 및 운용**
   - 키는 HSM 외부로 평문(plaintext) 반출을 금지한다. 키 래핑(key wrapping, AES-GCM 또는 RSA-OAEP) 방식으로만 대상 시스템에 배포한다.
   - 키별 접근 권한(ACL)을 최소권한 원칙에 따라 설정하며, 특정 서비스 계정·IP 범위로만 접근을 제한한다.
   - 운용 중 키 접근 이벤트(읽기·복호화·서명 등)가 감사 로그에 기록되며, SecOps가 이상 접근 패턴을 실시간 탐지한다.
3. **키 교체 (로테이션)**
   - 로테이션 주기: 대칭키(AES) 1년, TLS 인증서 키 1년(최대 397일), 루트 CA 키 3년, 코드 서명 키 2년. KMS Admin이 만료 60일 전 자동 알림을 수신한다.
   - [[TMP-KMS-003_키_로테이션_체크리스트_v0.1]] 를 사용하여 로테이션 전·후를 단계별로 검증한다.
   - 새 키 생성 → 대상 시스템에 새 키 배포 → 구 키로 암호화된 데이터 재암호화(필요 시) → 구 키 비활성화(deactivation) → 30일 유예 후 폐기(destruction) 순서로 진행한다.
   - 로테이션 완료 후 인벤토리 대장을 업데이트하고 오동석에게 완료 보고를 제출한다.
4. **키 보관 (장기 보존키)**
   - 법적 보존 의무가 있는 데이터 복호화에 사용된 키는 데이터 보존 기간 + 2년간 보존한다.
   - 보존키는 별도 키 보관 영역(Key Archive Partition)에 이동하고, 접근 시 오동석 승인 + Dual Control을 적용한다.
5. **키 폐기**
   - 유효기간 만료, 침해 의심, 서비스 종료 시 키를 폐기한다.
   - HSM 내 키 폐기는 Zeroization(모든 키 소재를 0으로 덮어쓰기) 방식을 사용하며, 폐기 완료 즉시 인벤토리 상태를 `destroyed` 로 변경한다.
   - 폐기 확인서(폐기 일시·방법·검증자 서명)를 작성하고 5년간 보관한다.
6. **HSM 보안 부팅 및 펌웨어 무결성 검증**
   - HSM 재부팅 후 보안 부팅 상태(Secure Boot ON, 서명 체인 유효)를 관리 콘솔에서 확인하고 기록한다.
   - 펌웨어 업데이트 시 제조사 제공 서명 파일을 검증한 후 적용하며, 적용 전·후 해시값을 기록한다.
7. **연 1회 알고리즘 적정성 검토**
   - KMS Admin이 현재 사용 중인 알고리즘 목록을 추출하고, NIST SP 800-57 Part 1 최신본 및 업계 권고를 참조하여 취약 알고리즘 여부를 검토한다.
   - 취약 알고리즘 발견 시 6개월 이내 교체 계획을 수립하고 오동석 승인을 받는다.

### 5.3 완료 조건

- [ ] 모든 운용 키가 [[TMP-KMS-001_암호키_인벤토리_대장_v0.1]] 에 등록되어 있고 상태가 최신화되었다
- [ ] 키 생성·배포·로테이션·폐기 이벤트가 [[TMP-KMS-002_HSM_접근_감사_로그_v0.1]] 에 기록되었다
- [ ] 로테이션 체크리스트 전 항목이 완료되었고 오동석 완료 보고가 제출되었다
- [ ] 폐기된 키의 폐기 확인서가 작성되고 보관되었다
- [ ] HSM 보안 부팅 상태가 재부팅 시마다 확인·기록되었다
- [ ] 연 1회 알고리즘 적정성 검토가 완료되었다
- [ ] 비상 복구 백업본의 위치·접근 권한자 목록이 최신화되었다

## 6. 인터페이스 부서

- **개발팀 / 서비스 소유자**: 키 사용 요구서 제출, 신규 서비스 키 배포 수신
- **인프라팀**: HSM 하드웨어 유지보수(전원·네트워크·물리 보안), 클라우드 HSM 계정 관리
- **SecOps**: 감사 로그 실시간 이상 탐지, 침해 의심 키 폐기 요청
- **오동석**: 키 정책 최종 승인, 예외 승인, 알고리즘 교체 계획 승인
- **내부감사팀**: 연 1회 HSM/KMS 운영 감사, 인벤토리·로그 대조 검증

## 7. 주의사항 / 예외 처리

### 7.1 키 침해 의심 시 긴급 폐기

침해 의심 징후(예: 권한 없는 키 접근, 시스템 침해 사고 연계)가 탐지되면, SecOps가 KMS Admin에게 즉시 통보하고 KMS Admin은 오동석 승인 없이 해당 키를 즉시 비활성화(emergency suspension)한다. 비활성화 후 2시간 이내 신규 키로 교체하고, 비활성화된 구 키의 폐기는 포렌식 조사 완료 후 오동석 승인으로 진행한다.

### 7.2 Dual Control 요건 미충족 시 (KMS Admin 부재)

KMS Admin이 단독으로만 접근 가능한 상황(다른 Admin 부재)에서는 긴급 키 작업이 불가하다. 이 경우를 대비해 백업 KMS Admin을 지정하고 긴급 접근 절차(Break Glass 프로세스)를 별도 문서화하며, 오동석만 백업 Admin 계정을 봉인 봉투 형태로 보관한다. 긴급 사용 후 즉시 비밀번호를 변경하고 사후 감사를 수행한다.

### 7.3 클라우드 관리형 KMS 사용 시

CSP 관리형 KMS(AWS KMS, Azure Key Vault 등) 사용 시에는 조직 소유 CMK(Customer Managed Key)를 사용하며, CSP의 기본 관리 키(AWS Managed Key 등)를 민감 데이터 암호화에 단독 사용하는 것을 금지한다. CMK 삭제 보호(Deletion Protection, 최소 30일 보호 기간)를 활성화하고, CloudTrail·Activity Log 등 CSP 감사 로그를 조직 SIEM에 연동한다.

### 7.4 HSM 하드웨어 장애

HSM 하드웨어 장애 발생 시 인프라팀이 장애 원인을 확인하고 KMS Admin에게 즉시 통보한다. 고가용성(HA) 구성의 보조 HSM으로 자동 전환되며, 보조 HSM 없는 경우 복구 절차(키 백업본 사용 또는 클라우드 HSM 임시 전환)를 RTO 4시간 이내 수행한다.
