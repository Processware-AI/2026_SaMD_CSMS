# 시스템 하드닝 업무지침 (WI-501-02)

> 상위 절차: [[PRO-MDCS-501_보안_모니터링_및_로그_관리_절차_v1.0]] · §5 단계 1 (공격 표면 최소화)

## 1. 업무 목적

OS·미들웨어·의료기기 펌웨어에 대해 불필요한 서비스·포트·인터페이스를 비활성화하고 보안 기준선(Security Baseline)을 적용함으로써 공격 표면을 최소화하며, 디지털의료기기의 사이버 보안 취약점이 실제 침해로 연결되는 경로를 사전에 차단한다.

## 2. 수행 주체

- **주 수행자**: 인프라팀 (하드닝 적용·기준선 유지)
- **검토자**: SecOps 팀장 (설정 적합성 검토), 개발팀 (의료기기 펌웨어 영향도 검토)
- **승인자**: 오동석 (하드닝 기준선 최초 승인 및 연간 재승인, 예외 승인)

## 3. 범위

- 운영 환경 서버 OS (Linux 계열, Windows Server)
- 웹 서버·WAS·데이터베이스 서버 등 미들웨어
- 의료기기 온보드 펌웨어 및 임베디드 OS (적용 가능한 범위)
- 클라우드 인스턴스 및 컨테이너 이미지 베이스
- 신규 시스템 배포 시 및 연간 정기 검토 시

## 4. 입력 자료 / 산출물

- **Input**
  - 자산 목록 (서버·장비 유형·OS 버전 포함)
  - CIS Benchmark 또는 조직 내 승인된 하드닝 기준선 문서
  - [[TMP-OPS-010_하드닝_체크리스트_v0.1]] (대상별 설정 항목)
  - 이전 취약점 스캔 결과 및 침투 테스트 보고서
- **Output**
  - 완료된 [[TMP-OPS-010_하드닝_체크리스트_v0.1]] (시스템별 설정 완료 증적)
  - 하드닝 예외 항목 목록 및 [[TMP-OPS-011_예외_승인서_v0.1]]
  - 취약점 스캔 결과 비교 보고서 (하드닝 전후)
  - 변경 이력 로그 (ITSM 변경 티켓 번호 포함)

## 5. 수행 절차

### 5.1 사전 준비

1. 인프라팀은 대상 시스템의 OS 유형·버전·역할을 자산 목록에서 확인하고, 적용할 하드닝 기준선(CIS Benchmark 또는 내부 승인 기준) 최신 버전을 확보한다.
2. 변경 적용 전 ITSM 시스템에 변경 요청(Change Request)을 등록하고, 변경 영향 범위·테스트 계획·롤백 방안을 기재한다.
3. 대상 시스템의 스냅샷 또는 이미지 백업을 생성하여 롤백 준비를 완료한다.
4. 의료기기 펌웨어 하드닝의 경우, 개발팀과 사전 협의하여 임상 기능 영향도 검토를 완료한다.

### 5.2 수행 단계

1. **불필요 서비스 및 포트 비활성화**
   - [[TMP-OPS-010_하드닝_체크리스트_v0.1]] 의 서비스 항목에 따라 사용하지 않는 네트워크 서비스(Telnet, FTP, SNMP v1/v2, rsh 등)를 비활성화·제거한다.
   - 방화벽·ACL 정책을 검토하여 업무 목적 외의 포트를 모두 닫고, 허용 포트 목록을 최소화한다.
   - 사용하지 않는 OS 기능·역할(Windows Server Role, Linux 데몬 등)을 제거한다.

2. **계정 및 인증 기준 강화**
   - 기본(default) 계정의 이름을 변경하거나 비활성화한다.
   - 패스워드 복잡도·최대 사용 기간·이전 비밀번호 재사용 금지 정책을 OS 수준에서 적용한다.
   - 원격 관리(SSH 등)는 키 기반 인증을 우선 적용하고, 루트(root) 직접 로그인을 비활성화한다.

3. **OS 및 미들웨어 보안 설정 적용**
   - SELinux 또는 AppArmor 등 강제 접근 통제(MAC) 모듈을 활성화한다.
   - 파일 시스템 마운트 옵션(`noexec`, `nosuid`, `nodev`)을 필요 파티션에 적용한다.
   - 웹 서버·데이터베이스 서버의 배너 정보 노출을 최소화하고, 관리 인터페이스 접근을 내부 네트워크로 제한한다.
   - 미들웨어 샘플·데모 애플리케이션 및 불필요한 모듈을 제거한다.

4. **의료기기 펌웨어 하드닝**
   - 개발팀이 제공한 펌웨어 하드닝 가이드에 따라 디버그 인터페이스(JTAG, UART 등)를 비활성화한다.
   - 불필요한 원격 관리 프로토콜 및 테스트 계정을 제거한다.
   - Secure Boot 또는 코드 서명 검증 기능이 활성화되어 있는지 확인한다.

5. **설정 검증 및 취약점 스캔**
   - 하드닝 완료 후 자동화된 설정 검사 도구(OpenSCAP, Lynis, CIS-CAT 등)로 기준선 준수율을 측정한다.
   - 취약점 스캐너(Nessus, OpenVAS 등)로 노출된 서비스·포트를 재확인한다.
   - 기준 미달 항목은 즉시 재조치하거나 예외 처리 절차(§7.1)를 따른다.

6. **예외 항목 처리 및 보완 통제 적용**
   - 업무상 비활성화가 불가한 서비스에 대해서는 [[TMP-OPS-011_예외_승인서_v0.1]]를 작성하고 오동석 승인을 받는다.
   - 예외 항목에 대해서는 접근 제어 강화·모니터링 강화·세션 타임아웃 등 보완 통제를 적용한다.

7. **변경 이력 기록 및 기준선 업데이트**
   - ITSM 변경 티켓을 완료(Close) 처리하고, 설정 변경 내역을 구성 관리 데이터베이스(CMDB)에 반영한다.
   - 하드닝 기준선 문서를 최신 상태로 유지하고, 연간 1회 이상 오동석 재승인을 받는다.

### 5.3 완료 조건

- [ ] 대상 시스템의 [[TMP-OPS-010_하드닝_체크리스트_v0.1]]가 100% 검토·완료되었다
- [ ] 불필요 서비스·포트가 비활성화되었고, 취약점 스캔 결과로 확인되었다
- [ ] 기본 계정 비활성화 및 패스워드 정책이 OS 수준에서 적용되었다
- [ ] 예외 항목이 [[TMP-OPS-011_예외_승인서_v0.1]]에 기재되고 오동석 승인을 받았다
- [ ] 하드닝 전후 취약점 스캔 결과 비교 보고서가 작성되었다
- [ ] ITSM 변경 티켓이 완료 처리되고 CMDB에 변경 내용이 반영되었다
- [ ] 의료기기 펌웨어 하드닝의 경우, 개발팀 임상 기능 영향 없음 확인서가 첨부되었다

## 6. 인터페이스 부서

- **SecOps 팀**: 하드닝 설정 적합성 검토, 이상 탐지 연계 (SIEM 룰 조정)
- **개발팀**: 의료기기 펌웨어 하드닝 가이드 제공, 임상 기능 영향 검토
- **오동석**: 하드닝 기준선 최초 승인 및 연간 재승인, 예외 승인
- **QA팀**: 하드닝 후 기능·성능 회귀 테스트 지원
- **변경관리위원회(CAB)**: 운영 환경 변경 승인

## 7. 주의사항 / 예외 처리

### 7.1 임상 기능에 영향을 주는 설정 변경

특정 서비스·포트가 의료기기의 임상 기능(환자 데이터 전송, 알람 연동 등)에 직접 연결된 경우, 단독으로 비활성화하지 않는다. 개발팀 및 RA(규제 담당)와 협의하여 임상 영향 없음을 확인한 후 적용하며, 확인 결과는 변경 티켓에 첨부한다. 불가피하게 서비스를 유지해야 하는 경우 §7.2 예외 처리 절차를 따른다.

### 7.2 업무상 불가피한 예외 서비스

레거시 의료 정보 시스템 연동, 규제 요건에 따른 특정 프로토콜 사용 등으로 취약한 서비스를 유지해야 하는 경우:
- [[TMP-OPS-011_예외_승인서_v0.1]]에 예외 사유, 잔존 위험, 보완 통제 내용을 상세 기재한다.
- 오동석 승인 후 예외 레지스트리에 등록하고, 6개월 이내 재검토 일정을 명시한다.
- 보완 통제로 접근 IP 화이트리스트 제한, 세션 로그 전량 수집, 이상 접근 알림을 반드시 적용한다.

### 7.3 클라우드 인스턴스 및 컨테이너 이미지

클라우드 인스턴스 하드닝은 IaC(Infrastructure as Code) 코드에 기준선 설정을 반영하여 이미지 빌드 시 자동 적용한다. 컨테이너 이미지는 Distroless 또는 최소 베이스 이미지를 사용하고, 루트 권한으로 실행하지 않는다. CI/CD 파이프라인에 CIS Docker Benchmark 검사를 통합하여 비준수 이미지 배포를 차단한다.

### 7.4 긴급 배포 시 하드닝 절차 단축

침해사고 대응 중 긴급 패치 배포로 하드닝 검증 단계를 단축해야 하는 경우, 최소한 "불필요 포트 차단" 및 "기본 계정 비활성화" 2개 항목은 반드시 적용한 후 배포한다. 단축된 항목은 72시간 이내 후속 점검을 완료하고 결과를 오동석에 보고한다.
