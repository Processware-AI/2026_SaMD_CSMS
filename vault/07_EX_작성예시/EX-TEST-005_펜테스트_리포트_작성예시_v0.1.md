## 5. 주요 발견 사항 요약

### 5.1 Critical / High 취약점 상세

#### 취약점 #1: JWT Algorithm None 인증 우회

| 항목 | 내용 |
|---|---|
| 취약점 ID | PEN-2026-002-001 |
| 심각도 | Critical |
| 영향 자산 | API 서버 /v2/* 엔드포인트 전체 |
| CVE | 해당 없음 (설정 오류) |
| 설명 | JWT 라이브러리에서 `alg: none` 을 명시적으로 거부하지 않아, 공격자가 서명 없이 임의 사용자 ID(sub)를 포함한 JWT를 생성하여 모든 환자 혈당 데이터에 접근 가능 |
| 재현 방법 | `{"alg":"none","typ":"JWT"}` 헤더 + `{"sub":"patient-9999"}` 페이로드로 서명 없는 JWT 생성 → `GET /v2/glucose/patient-9999` 요청 → 200 OK + 혈당 데이터 반환 확인 |
| 조치 권고 | PyJWT 라이브러리 `algorithms=["HS256"]` 화이트리스트 설정; `options={"verify_signature": True}` 강제 |

#### 취약점 #2: BLE 재전송 공격

| 항목 | 내용 |
|---|---|
| 취약점 ID | PEN-2026-002-002 |
| 심각도 | High |
| 영향 자산 | CGM 센서 ↔ 앱 BLE 구간 |
| CVE | 해당 없음 |
| 설명 | BLE 혈당 패킷에 단조 증가 카운터 또는 타임스탬프 검증이 없어, 이전에 캡처한 패킷을 재전송하면 앱이 오래된 혈당 값을 유효한 측정값으로 처리함 |
| 재현 방법 | nRF Sniffer로 정상 BLE 패킷 캡처 → 30분 후 동일 패킷 재전송 → 앱 UI에 과거 혈당 값 표시 확인 |
| 조치 권고 | 패킷 내 단조 증가 시퀀스 번호(nonce) 및 타임스탬프 추가; 앱 측 5분 이상 경과 패킷 거부 로직 구현 |

## 작성 요령

- **테스트 유형**: 블랙박스/화이트박스/그레이박스 중 어떤 정보를 제공받았는지 명기한다. 제공 정보(API 문서, 계정 등)를 기재하면 테스트 범위의 충분성 판단에 도움이 된다.
- **취약점 목록**: 발견 건수가 0이어도 테스트를 수행했다면 '해당 없음'으로 기재한다. Informational급도 포함하여 기록하되, 우선순위 조치 표에서는 High 이상을 앞에 배치한다.
- **주요 발견 사항 상세**: Critical/High 취약점은 재현 방법을 단계별로 기재하되, 악용 가능한 구체적 페이로드는 보고서 수신자 범위를 제한하여 관리한다.
- **권고 기한**: '즉시', '가능한 빨리' 같은 모호한 표현 대신 구체적인 날짜를 기재한다. Critical은 24~72시간 내 조치가 원칙이다.
