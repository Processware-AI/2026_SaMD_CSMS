## 작성 요령

- **단계별 완료 일시**: 각 단계의 완료 일시를 정확히 기재하면 나중에 문제가 발생했을 때 어느 단계에서 이슈가 생겼는지 추적할 수 있습니다.
- **담당자 구분**: 단계별 담당자를 명확히 구분합니다. 특히 신규 키 생성, 백업 검증, 폐기 처리는 가능하면 서로 다른 담당자가 수행하여 이중 확인(4-eyes) 원칙을 적용합니다.
- **비고란 활용**: 백업 스킴(N-of-M), 오프사이트 보관 여부, 재암호화 대상 시스템 수 등을 비고란에 기재하면 감사 추적성이 높아집니다.
- **로테이션 사유**: 주기도래, 보안사고, 정책변경 중 해당 항목을 명확히 선택하고, 관련 근거 문서(정책 번호 등)를 링크합니다.

## 잘못된 기재 사례

### NG 사례 1 — "구 키 폐기" 단계 없이 로테이션 완료 처리

| 단계 | 작업 항목 | 완료 여부 |
|---|---|---|
| 4. 구 키 폐기 | 기존 키 비활성화 | **미완료** |
| 4. 구 키 폐기 | 기존 키 안전 삭제·파기 | **미완료** |

- **문제**: 구 키를 폐기하지 않고 로테이션 완료로 처리하면 구 키가 여전히 노출 위험에 있어 보안 사고로 이어질 수 있습니다.
- **올바른 예시**: 신규 키 적용 안정화 확인 후 지정 보관 기간 내에 반드시 구 키를 Zeroize 처리하고, 완료 일시와 처리자를 기재합니다.

### NG 사례 2 — 실제 완료일 없이 "완료 목표일"만 기재

- **문제**: 완료 목표일만 있고 실제 완료일을 공란으로 두면 실제 수행 여부 확인이 불가능합니다.
- **올바른 예시**: 모든 단계 완료 후 실제 완료일을 반드시 기재하고 승인자 서명을 득합니다.
