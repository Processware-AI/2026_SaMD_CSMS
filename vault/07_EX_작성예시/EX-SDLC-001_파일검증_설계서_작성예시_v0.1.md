# 파일검증 설계서 작성예시 (EX-SDLC-001)

> 원본 양식: [[TMP-SDLC-001_파일검증_설계서_v0.1]]

> **이 문서는 교육용 예시로, 실제 기록(REC)이 아닙니다.**

---

## 1. 설계 기본 정보

| 항목 | 내용 |
|---|---|
| 대상 기능 | MediGuard CGM App v2.3.0 — 펌웨어 OTA 업로드 기능 (`POST /v2/ota/upload`) |
| 설계자 | 이개발 (Lead Dev, Backend) |
| 검토자 | 김보안 (PSO) |
| 설계일 | 2026-03-15 |

## 2. 파일 검증 정책

| 항목 | 내용 |
|---|---|
| 허용 파일 유형 | `.fw` (펌웨어 바이너리) 단일 유형만 허용 |
| 최대 크기 | 10 MB (10,485,760 바이트) |
| 검증 방식 | ① 확장자 검증 (`.fw`만 허용) → ② 매직넘버 검증 (0x4D47 헤더 확인) → ③ RSA-PSS 서명 검증 (공개키: OTA-PUBKEY-2026-01) → ④ SHA-256 해시 무결성 확인 → ⑤ ClamAV 바이러스 스캔 |
| 차단 응답 정책 | HTTP 422 Unprocessable Entity 반환, 오류 메시지는 `"Invalid firmware package"` 단일 메시지 (세부 오류 비노출), 서버 측 로그에 상세 사유 기록 |

---

## 작성 요령

- **대상 기능**: API 엔드포인트 경로와 HTTP 메서드를 구체적으로 기재한다. '파일 업로드 기능'처럼 모호하게 기재하면 구현·검증 연결이 어렵다.
- **허용 파일 유형**: 확장자만 검증하면 파일 유형 위장 공격에 취약하다. 매직넘버(파일 시그니처) 검증을 반드시 병행하고 두 방식을 모두 기재한다.
- **검증 방식**: 검증 순서(확장자 → 매직넘버 → 서명 → 무결성 → AV)를 명시하고, 각 단계에서 실패 시 이후 단계를 수행하지 않아야 함을 설계서에 명기한다.
- **차단 응답 정책**: 오류 메시지에 내부 경로·기술 정보를 노출하면 공격자 정보 수집에 악용된다. 사용자 표시 메시지와 서버 로그 메시지를 분리하여 기재한다.

## 잘못된 기재 사례

### NG 사례 — 확장자 검증만 적용·상세 오류 노출

| 항목 | 내용 |
|---|---|
| 허용 파일 유형 | .fw |
| 최대 크기 | 10MB |
| 검증 방식 | 확장자 |
| 차단 응답 정책 | `"Error: invalid extension, expected .fw, got .exe at /app/upload/handler.py:142"` |

**문제점**: 확장자 검증만으로는 `.fw`로 이름을 바꾼 악성 파일을 차단할 수 없다. 매직넘버·서명 검증 누락. 차단 응답 메시지에 내부 파일 경로(`/app/upload/handler.py:142`)와 라인 번호가 노출되어 공격자가 서버 구조 파악 가능.

**올바른 예시**: 매직넘버·RSA-PSS 서명·AV 스캔을 검증 방식에 추가하고, 차단 응답은 `"Invalid firmware package"` 단일 메시지, 세부 정보는 서버 로그에만 기록.
