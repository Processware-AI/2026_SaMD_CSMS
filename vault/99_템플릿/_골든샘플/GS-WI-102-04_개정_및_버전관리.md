# 개정 및 버전관리 업무지침 (GS-WI-102-04)

> **[골든 샘플]** 이 문서는 `WI` 유형의 품질 하한선 참조용입니다. 실제 산출물 아님.
> 상위 절차: [[GS-PRO-QMS-102_문서_개정_관리_절차]]

## 1. 업무 목적
승인된 개정 요청을 문서에 반영하고 최신판으로 배포하여, 조직 전체가 동일한 최신 기준으로 업무를 수행하도록 한다.

## 2. 수행 주체
- **주 수행자**: 해당 문서의 작성자(Author)
- **검토자**: Process Owner
- **승인자**: QMR (경미) / CEO (중대 시, GS-PRO-QMS-102 에 위임)

## 3. 범위
본 지침은 GS-PRO-QMS-102 §5 의 **Step 6 반영 ~ Step 7 배포·기록** 단계에 적용한다. 요청 접수·영향평가·심의는 별도 WI(-01, -02) 참조.

## 4. 입력 자료 / 산출물
- **Input**
  - 승인된 [[TMP-DOC-001_개정요청서]] (경미/중대 분류 포함)
  - 필요 시 [[TMP-DOC-002_영향평가서]]
  - 현 승인판 문서(vX.Y)
- **Output**
  - 개정본 문서(vX.(Y+1) 또는 v(X+1).0)
  - [[TMP-DOC-003_배포공지문]] 작성분
  - [[MAT-001_문서관리대장]] 신규 Row 등록

## 5. 수행 절차

### 5.1 사전 준비
1. 현 승인판 파일 백업: `vault/99_폐기_보관/{원본파일명}_v{현재}_archived_{YYYYMMDD}.md`
2. 개정요청서의 "변경 요지" 확인·요약.
3. 경미/중대 분류 재확인 (변경 범위가 당초보다 크면 GS-PRO-QMS-102 §5 Step 4 로 에스컬레이션).

### 5.2 수행 단계
1. **개정본 작성**
   - 본문 수정, 변경된 부분에 내부 주석(`<!-- changed: ... -->`) 임시 삽입
   - frontmatter 의 `version` 증분:
     - 경미: `v1.2` → `v1.3`
     - 중대: `v1.3` → `v2.0`
   - frontmatter `updated` 를 오늘 날짜로
2. **내부 링크 검증**
   - Grep 으로 구버전 파일명 참조 탐색 → 신버전 파일명으로 일괄 Edit
3. **개정 이력 섹션 추가**
   - 문서 말미 `개정 이력` 표에 Row 추가:
     - 버전 / 일자 / 변경내용(1~2줄) / 승인자
4. **검토 승인**
   - Process Owner 리뷰 코멘트 반영
   - 승인자에게 승인 요청(전자결재 또는 PCB 회의록)
   - 승인 후 내부 주석(`<!-- changed: ... -->`) 제거
5. **파일 저장·커밋**
   - 경로는 변경하지 않음 (파일명은 버전에 무관한 안정 식별자)
   - Git 운용 시: `docs: revise {doc_id} to v{new_version}` 로 커밋
6. **배포 공지**
   - [[TMP-DOC-003_배포공지문]] 작성: 변경 요지 3줄 + 적용일 + 관련자 태그
   - 사내 공지채널 게시
7. **문서관리대장 등록**
   - [[MAT-001_문서관리대장]] 에 Row 추가 또는 기존 Row 갱신:
     - `version`·`updated`·`status=approved` 필드

### 5.3 완료 조건
- [ ] 개정본 frontmatter 의 version·updated 갱신 완료
- [ ] 개정 이력 표 Row 추가 완료
- [ ] 관련 내부 링크(`[[...]]`) 무결성 확인
- [ ] 배포 공지 게시
- [ ] MAT-001 등록 확인

## 6. 인터페이스 부서
- **QMR**: 경미 개정 승인 기록 보관
- **IT팀**: Git/EDMS 저장소 운영
- **커뮤니케이션팀**: 사내 공지 채널

## 7. 주의사항 / 예외 처리

### 7.1 긴급 개정 (Hot-fix)
- 법규·보안 이슈로 즉시 반영 필요 시:
  - QMR 구두 승인 → 선반영·선배포
  - 48시간 이내 정식 결재 보완
  - 사후 PCB 정기회의에 보고

### 7.2 전면 재작성
- 본문 50% 이상 변경 또는 `scope`/`owner` 변경 시:
  - 개정이 아닌 **신규 문서 생성** 으로 처리 (신규 doc_id 부여)
  - 구 문서는 `status: superseded` + `superseded_by: [[신규문서]]` 메타로 변경
  - `99_폐기_보관/` 로 이동하지 않음 (이력 보존 목적)

### 7.3 버전 번호 충돌
- 동시 개정으로 버전 번호가 중복된 경우:
  - 후행 작성자가 +1 증분하여 회피
  - MAT-001 의 `updated` 타임스탬프로 순서 확정
